# FullCon Use Cases

작성일: 2026-05-01

이 문서는 [00-master-spec.md](./00-master-spec.md)를 기준으로 실제 구현해야 하는 유즈케이스를 분리 정리한 문서다.

원칙:

- 이 문서는 정본 스펙이 아니라 실행용 파생 문서다
- 규칙 충돌 시 `00-master-spec.md`를 우선한다
- 각 유즈케이스는 `행위자`, `트리거`, `선행조건`, `메인 흐름`, `예외`, `결과` 기준으로 본다
- 현재 메인 제품에 포함되지 않는 레거시 기능은 유즈케이스로 유지하지 않는다
- 레거시 데이터가 남아 있어도 현재 제품을 위해 필요한 경우에만 `호환 전용 내부 로직`으로 표현한다

## 1. 온보딩

### UC-01 첫 실행 온보딩 시작

- 행위자: 신규 사용자
- 트리거: 앱 첫 실행
- 선행조건: `isCompleted = false`
- 메인 흐름:
  1. 앱이 `/onboarding`으로 진입한다
  2. 제품 소개와 시작 CTA를 보여준다
  3. 사용자가 온보딩을 시작한다
- 예외:
  - 온보딩 데이터 일부가 있으면 마지막 안전 단계부터 복구한다
- 결과:
  - 사용자는 닉네임 입력 또는 첫 체크인 단계로 이동한다

### UC-02 닉네임 입력

- 행위자: 신규 사용자
- 트리거: 온보딩 identity 단계 진입
- 선행조건: 온보딩 진행 중
- 메인 흐름:
  1. 사용자가 닉네임을 입력한다
  2. 시스템이 trim과 빈 값 검사를 한다
  3. 다음 단계로 이동한다
- 예외:
  - 빈 닉네임이면 진행하지 않는다
- 결과:
  - 온보딩 draft에 닉네임이 저장된다

### UC-03 첫 체크인 완료로 온보딩 종료

- 행위자: 신규 사용자
- 트리거: 온보딩 first check-in 단계 저장
- 선행조건: 닉네임 저장 완료
- 메인 흐름:
  1. 사용자가 기본 체크인을 완료한다
  2. 시스템이 `ConditionRecord`를 저장한다
  3. 시스템이 `isCompleted = true`로 업데이트한다
  4. 홈으로 이동한다
- 예외:
  - 저장 실패 시 온보딩 완료 처리하지 않는다
- 결과:
  - 사용자는 홈에 진입한다

### UC-04 온보딩 중단 후 복귀

- 행위자: 신규 사용자
- 트리거: 앱 재실행 또는 딥링크 진입
- 선행조건: 온보딩 미완료
- 메인 흐름:
  1. 시스템이 온보딩 draft를 확인한다
  2. 마지막 안전 단계부터 재개한다
- 예외:
  - draft가 손상되었으면 해당 단계 처음부터 재시작한다
- 결과:
  - 사용자는 처음부터 다시 갇히지 않고 복귀한다

## 2. 홈과 코어 루프

### UC-05 홈 진입 시 오늘 상태 확인

- 행위자: 온보딩 완료 사용자
- 트리거: 홈 진입 또는 홈 복귀
- 선행조건: `isCompleted = true`
- 메인 흐름:
  1. 시스템이 최신 `ConditionRecord`, 오늘 `SleepRecord`, 최근 7일 기록을 읽는다
  2. 상태 헤드라인을 계산한다
  3. 메인 CTA 1개를 계산한다
  4. `오늘 상태`, `지금 할 일`, `이번 주 한 줄`을 보여준다
- 예외:
  - 데이터가 없으면 empty 상태와 첫 행동 CTA를 보여준다
- 결과:
  - 사용자는 지금 필요한 행동 1개를 본다

### UC-06 오늘 첫 체크인 유도

- 행위자: 사용자
- 트리거: 홈에서 오늘 체크인이 없을 때
- 선행조건: 당일 체크인 없음
- 메인 흐름:
  1. 홈 메인 CTA가 `오늘 체크인`으로 계산된다
  2. 사용자가 CTA를 누른다
  3. 체크인 화면으로 이동한다
- 예외:
  - 사용자가 CTA를 무시하면 추가 표면 노출 규칙에 따라 나중에 다시 제안한다
- 결과:
  - 체크인 플로우가 시작된다

### UC-07 회복 로그 유도

- 행위자: 사용자
- 트리거: 회복 데이터가 없거나 낮을 때
- 선행조건: 홈 상태 계산 완료
- 메인 흐름:
  1. 시스템이 회복 부족 또는 데이터 없음 상태를 감지한다
  2. 홈 메인 CTA를 `회복 로그 남기기`로 계산한다
  3. 사용자가 CTA를 눌러 수면 기록 화면으로 이동한다
- 예외:
  - 이미 최근 회복 로그가 있으면 다른 CTA를 우선한다
- 결과:
  - 회복 로그 플로우가 시작된다

### UC-08 데이터 신선도 저하 알림

- 행위자: 시스템
- 트리거: 오늘 체크인 없음, 최근 회복 로그 없음, 신호 부족
- 선행조건: 홈 또는 개입 평가 시점
- 메인 흐름:
  1. 시스템이 stale data를 감지한다
  2. 홈 배너, CTA, micro check-in 중 적절한 표면을 선택한다
  3. 사용자에게 정확도 저하와 필요한 다음 입력을 제안한다
- 예외:
  - quiet hours 또는 active focus block이면 일반 개입을 억제한다
- 결과:
  - 사용자는 데이터 보강 행동을 할 수 있다

## 3. 컨디션 체크인

### UC-09 기본 체크인 저장

- 행위자: 사용자
- 트리거: 체크인 화면 진입 후 저장 버튼 탭
- 선행조건: 기본 점수 입력 가능 상태
- 메인 흐름:
  1. 사용자가 에너지와 회복을 입력한다
  2. 필요 시 집중, 스트레스, 태그, 상황, 몸 반응, 메모를 추가한다
  3. 시스템이 값 정규화와 유효성 검사를 한다
  4. `ConditionRecord`를 저장한다
  5. 홈을 refresh 한다
- 예외:
  - 저장 실패 시 입력값은 유지한 채 오류를 보여준다
- 결과:
  - 새로운 체크인 기록이 저장된다

### UC-10 마이크로 체크인 응답

- 행위자: 사용자
- 트리거: micro check-in 노출
- 선행조건: 상태 불확실 또는 데이터 충돌
- 메인 흐름:
  1. 시스템이 1-tap 응답 선택지를 보여준다
  2. 사용자가 하나를 선택한다
  3. 시스템이 상태 추정과 개입 규칙을 갱신한다
- 예외:
  - 사용자가 무시하면 outcome을 `ignored` 또는 `dismissed`로 남긴다
- 결과:
  - 짧은 self-report가 상태 판정에 반영된다

### UC-11 체크인 수정 또는 삭제

- 행위자: 사용자
- 트리거: 기존 체크인 상세 또는 타임라인 진입
- 선행조건: 기존 체크인 존재
- 메인 흐름:
  1. 사용자가 수정 또는 삭제를 선택한다
  2. 시스템이 저장 또는 삭제를 수행한다
  3. 홈, 타임라인, 집계를 다시 계산한다
- 예외:
  - 삭제 취소 시 원상태 유지
- 결과:
  - 기록과 집계가 최신 상태가 된다

## 4. 회복 로그

### UC-12 수면 기록 저장

- 행위자: 사용자
- 트리거: 회복 로그 화면 저장
- 선행조건: 취침/기상 시간 입력 가능
- 메인 흐름:
  1. 사용자가 취침 시간, 기상 시간, 회복감, 수면 질을 입력한다
  2. 시스템이 cross-day 시간 해석과 파생값 계산을 한다
  3. `SleepRecord`를 저장한다
  4. 현재 화면을 닫고 홈/타임라인에 반영한다
- 예외:
  - 저장 실패 시 입력값 유지 + 오류 메시지
- 결과:
  - 회복 로그가 저장된다

### UC-13 수면 기록 수정

- 행위자: 사용자
- 트리거: 기존 회복 로그 편집
- 선행조건: 기존 `SleepRecord` 존재
- 메인 흐름:
  1. 화면이 기존 값을 hydrate 한다
  2. 사용자가 값을 수정한다
  3. 시스템이 기존 `id`를 유지한 채 저장한다
  4. 관련 집계를 갱신한다
- 예외:
  - 잘못된 시간 해석은 정규화 규칙으로 보정한다
- 결과:
  - 수면 기록이 수정된다

### UC-14 취침 전 wind-down 개입

- 행위자: 시스템, 사용자
- 트리거: 취침 전 위험 시점 감지
- 선행조건: quiet hours 직전 또는 취침 지연 가능성
- 메인 흐름:
  1. 시스템이 취침 전 개입을 노출한다
  2. 사용자는 `지금 자러 가기`, `15분 후`, `미루기` 중 하나를 선택한다
  3. 시스템이 선택 결과를 outcome으로 기록한다
  4. 필요 시 실제 취침 기록 입력으로 연결한다
- 예외:
  - 같은 세션에서 중복 취침 개입을 띄우지 않는다
- 결과:
  - 취침 행동 또는 취침 미루기 상태가 기록된다

## 5. 집중 블록

### UC-15 집중 블록 시작

- 행위자: 사용자
- 트리거: `집중 시작` CTA 탭
- 선행조건: 홈 또는 인앱 표면에서 집중 제안 노출
- 메인 흐름:
  1. 사용자가 집중 블록을 시작한다
  2. 시스템이 active focus block 상태로 전환한다
  3. 일반 proactive 질문을 억제한다
  4. 타이머 또는 세션 상태를 보여준다
- 예외:
  - 사용자가 바로 종료하면 짧은 outcome만 기록한다
- 결과:
  - 집중 세션이 활성화된다

### UC-16 집중 블록 중간 확인

- 행위자: 시스템, 사용자
- 트리거: 집중 블록 중간 시점
- 선행조건: active focus block
- 메인 흐름:
  1. 시스템이 1회 제한의 중간 확인을 노출한다
  2. 사용자가 상태를 탭으로 응답한다
  3. 시스템이 남은 세션 유지 또는 조정 제안을 한다
- 예외:
  - 응답하지 않아도 세션은 유지된다
- 결과:
  - 집중 지속 가능성 데이터가 쌓인다

### UC-17 집중 블록 종료

- 행위자: 사용자
- 트리거: 타이머 종료 또는 수동 종료
- 선행조건: active focus block
- 메인 흐름:
  1. 시스템이 종료 피드백을 요청한다
  2. 사용자가 완료 / 중단 / 미완료 중 하나를 선택한다
  3. 시스템이 outcome과 짧은 feedback을 기록한다
- 예외:
  - 피드백이 없어도 최소 outcome은 기록한다
- 결과:
  - 집중 블록 결과가 저장된다

## 6. 개입 시스템

### UC-18 proactive 개입 노출

- 행위자: 시스템
- 트리거: 상태 악화, 회복 부족, 취침 지연, 데이터 불확실
- 선행조건: 개입 budget 허용, suppression 없음
- 메인 흐름:
  1. 시스템이 개입 필요 여부를 계산한다
  2. 홈 CTA, 인앱, 알림 중 표면을 선택한다
  3. `InterventionEvent`를 기록한다
  4. 사용자 반응을 기다린다
- 예외:
  - quiet hours, dismiss 누적, active focus block이면 억제한다
- 결과:
  - 개입이 노출된다

### UC-19 개입 snooze

- 행위자: 사용자
- 트리거: 개입 표면에서 `10분 후`, `오늘 저녁`, `오늘은 그만` 선택
- 선행조건: 노출된 개입 존재
- 메인 흐름:
  1. 사용자가 snooze 또는 suppression 옵션을 고른다
  2. 시스템이 같은 목적 개입 재노출을 억제한다
  3. 결과를 이벤트로 남긴다
- 예외:
  - safety 대응 개입은 일반 snooze 규칙보다 우선할 수 있다
- 결과:
  - 당일 개입 budget 또는 시점이 조정된다

### UC-20 개입 완료 기록

- 행위자: 사용자, 시스템
- 트리거: 개입 실행 플로우 완료
- 선행조건: 노출된 개입을 사용자가 수락
- 메인 흐름:
  1. 실행 결과를 `completed`, `dismissed`, `ignored`, `snoozed` 등으로 정리한다
  2. 상태 변화가 있으면 `improved`, `unchanged`, `worsened`, `unknown`으로 기록한다
  3. 동일 목적 후속 개입 budget을 조정한다
- 예외:
  - 피드백이 없더라도 outcome은 기록한다
- 결과:
  - 개인화 학습에 사용할 결과 데이터가 남는다

### UC-21 안전 우선 개입

- 행위자: 시스템
- 트리거: 위험 신호 의심
- 선행조건: 안전 우선순위 발동
- 메인 흐름:
  1. 시스템이 일반 코칭을 중지한다
  2. 중립적이고 축소된 응답을 우선 노출한다
  3. 같은 세션에서 일반 개입을 추가로 겹치지 않는다
- 예외:
  - quiet hours와 focus suppression보다 안전 대응이 우선한다
- 결과:
  - 안전 대응 세션으로 처리된다

## 7. 타임라인과 회고

### UC-22 특정 날짜 기록 조회

- 행위자: 사용자
- 트리거: 월간 캘린더 날짜 선택
- 선행조건: 타임라인 화면 진입
- 메인 흐름:
  1. 시스템이 해당 날짜의 condition + sleep 기록을 조회한다
  2. 내부적으로 단일 엔트리 흐름으로 래핑한다
  3. 리스트와 요약을 보여준다
- 예외:
  - 기록이 없으면 empty 상태 문구를 보여준다
- 결과:

### UC-23 주간 요약 확인

- 예외:

### UC-24 인사이트 확신도 낮음 처리

## 10. 추가로 필요한 운영 유즈케이스

아래 유즈케이스는 현재 문서만으로도 방향은 유추 가능하지만, 실제 앱이 덜 흔들리게 하려면 별도로 명시하는 편이 좋다.

### UC-32 알림 탭으로 해당 개입 플로우 진입

- 행위자: 사용자
- 트리거: 시스템 알림 탭
- 선행조건: 노출된 개입 알림 존재
- 메인 흐름:
  1. 사용자가 알림을 탭한다
  2. 시스템이 해당 `correlationId`와 개입 목적을 찾는다
  3. 홈, 실행 화면, 체크인, 회복 로그 중 맞는 진입점으로 이동한다
  4. 중복 알림 상태를 정리한다
- 예외:
  - 대상 개입이 이미 완료되었으면 홈 또는 관련 기록 화면으로 보낸다
- 결과:
  - 알림과 실제 실행 플로우가 끊기지 않는다

### UC-33 권한 point-of-use 요청

- 행위자: 시스템, 사용자
- 트리거: 알림, 캘린더, 워치, 폰 사용 데이터가 실제로 필요한 시점
- 선행조건: 관련 권한 미허용
- 메인 흐름:
  1. 시스템이 왜 이 권한이 지금 필요한지 짧게 설명한다
  2. 사용자가 허용 또는 건너뛰기를 선택한다
  3. 허용 시 시스템 권한 요청으로 이어진다
  4. 거부 시 degraded mode로 계속 진행한다
- 예외:
  - permanently denied면 시스템 설정 이동만 제안한다
- 결과:
  - 권한 요청이 기능 흐름 안에서 자연스럽게 처리된다

### UC-34 체크인 draft 복구 또는 폐기

- 행위자: 사용자
- 트리거: 체크인 작성 중 앱 종료 후 재진입
- 선행조건: 미저장 체크인 draft 존재
- 메인 흐름:
  1. 시스템이 draft 존재를 확인한다
  2. 사용자가 이어서 작성 또는 버리기를 선택한다
  3. 이어서 작성 시 이전 입력을 hydrate 한다
  4. 버리기 시 draft를 삭제한다
- 예외:
  - draft가 오래되었거나 손상되면 폐기 후 새로 시작한다
- 결과:
  - 미완료 입력 때문에 저장/집계가 흔들리지 않는다

### UC-35 수면 기록 draft 복구 또는 폐기

- 행위자: 사용자
- 트리거: 수면 기록 작성 중 앱 종료 후 재진입
- 선행조건: 미저장 회복 로그 draft 존재
- 메인 흐름:
  1. 시스템이 draft 존재를 확인한다
  2. 사용자가 이어서 작성 또는 버리기를 선택한다
  3. 이어서 작성 시 시간과 입력값을 복구한다
- 예외:
  - 날짜 경계가 바뀌었으면 현재 날짜 기준으로 재확인한다
- 결과:
  - 수면 입력 중단이 기록 손실이나 잘못된 저장으로 이어지지 않는다

### UC-36 중복 저장 방지

- 행위자: 사용자, 시스템
- 트리거: 저장 버튼 연속 탭 또는 동일 요청 재전송
- 선행조건: 저장 진행 중
- 메인 흐름:
  1. 시스템이 첫 저장 요청을 잠근다
  2. 추가 탭이나 중복 요청을 무시하거나 병합한다
  3. 저장 결과 1회만 반영한다
- 예외:
  - 저장 실패 후에는 명시 재시도를 허용한다
- 결과:
  - 중복 기록과 중복 이벤트가 생기지 않는다

### UC-37 snooze 만료 후 개입 재평가

- 행위자: 시스템
- 트리거: `10분 후` 또는 예약된 snooze 만료
- 선행조건: snooze 상태의 개입 존재
- 메인 흐름:
  1. 시스템이 snooze 만료 시점에 현재 상태를 다시 평가한다
  2. 같은 목적 개입이 아직 유효한지 확인한다
  3. 필요하면 더 약한 표면 또는 같은 표면으로 재노출한다
- 예외:
  - 사용자가 이미 수동 행동을 완료했으면 개입을 폐기한다
- 결과:
  - snooze가 단순 지연이 아니라 재판단으로 이어진다

### UC-38 타임라인 월 이동과 빈 월 처리

- 행위자: 사용자
- 트리거: 이전/다음 월 탐색
- 선행조건: 타임라인 화면 진입
- 메인 흐름:
  1. 사용자가 월을 이동한다
  2. 시스템이 해당 월의 condition/sleep 기록을 다시 조회한다
  3. 기록이 있으면 날짜 마커와 리스트를 보여준다
  4. 기록이 없으면 빈 월 상태를 보여준다
- 예외:
  - 로딩 실패 시 재시도 액션을 제공한다
- 결과:
  - 월 탐색이 안정적으로 동작한다

### UC-39 앱 재진입 시 pending refresh 정리

- 행위자: 시스템
- 트리거: 앱 foreground 복귀 또는 cold start
- 선행조건: pending refresh, stale cache, failed event write 존재 가능
- 메인 흐름:
  1. 시스템이 미완료 refresh 플래그를 확인한다
  2. 홈/타임라인 집계를 다시 계산한다
  3. 실패했던 이벤트 write가 있으면 재시도한다
- 예외:
  - 원본 기록이 이미 삭제되었으면 orphan event write를 폐기한다
- 결과:
  - 재진입 후 화면과 저장 상태가 다시 맞춰진다

### UC-40 `왜 이 제안인가` 확인

- 행위자: 사용자
- 트리거: 개입 카드 또는 CTA의 설명 진입
- 선행조건: 개입 노출 중
- 메인 흐름:
  1. 사용자가 제안 이유를 확인한다
  2. 시스템이 최근 상태와 맥락 기준의 짧은 설명을 보여준다
  3. 사용자는 수락, 미루기, 닫기 중 하나를 선택한다
- 예외:
  - 설명 가능성이 낮으면 일반화된 짧은 이유만 보여준다
- 결과:
  - 개입 신뢰도가 올라가고 감시감이 줄어든다

## 11. 추가 엣지/반복 사용 유즈케이스

### UC-41 하루에 여러 번 체크인

- 행위자: 사용자
- 트리거: 이미 당일 체크인이 있는 상태에서 다시 체크인 시작
- 선행조건: 당일 `ConditionRecord` 1건 이상 존재
- 메인 흐름:
  1. 사용자가 다시 체크인 화면으로 들어간다
  2. 시스템이 새 기록으로 저장할지 최근 기록을 수정할지 정책에 따라 처리한다
  3. 새 기록이면 append 한다
  4. 홈은 가장 최신 체크인을 기준으로 다시 계산한다
- 예외:
  - 매우 짧은 시간 내 중복 입력은 중복 저장 방지 규칙을 우선 적용한다
- 결과:
  - 하루 중 상태 변화가 여러 번 기록될 수 있다

### UC-42 최근 체크인과 외부 신호 충돌

- 행위자: 시스템
- 트리거: self-report와 passive signal이 상충함
- 선행조건: 최근 수동 체크인 존재, 외부 신호 입력 가능
- 메인 흐름:
  1. 시스템이 충돌을 감지한다
  2. 수동 체크인을 우선 근거로 사용한다
  3. 필요 시 micro check-in으로 불확실성을 줄인다
- 예외:
  - 외부 신호가 없으면 self-report만 사용한다
- 결과:
  - 상태 판정이 수동 입력 우선 원칙을 따른다

### UC-43 홈에서 CTA 실행 후 즉시 복귀

- 행위자: 사용자
- 트리거: 홈 CTA 실행 후 중간에 뒤로가기 또는 닫기
- 선행조건: 실행 플로우 시작
- 메인 흐름:
  1. 사용자가 실행 플로우를 시작한다
  2. 중간에 홈으로 돌아간다
  3. 시스템이 `started but not completed` 상태를 남긴다
  4. 홈은 같은 목적 CTA를 바로 중복 노출하지 않고 재평가한다
- 예외:
  - 실제 완료 신호가 있으면 completed로 승격한다
- 결과:
  - 실행 중 이탈이 결과 데이터로 남는다

### UC-44 포커스 세션 중 앱 강제 종료 후 복귀

- 행위자: 사용자, 시스템
- 트리거: active focus block 중 앱 종료 후 재실행
- 선행조건: 종료 시점에 focus block active
- 메인 흐름:
  1. 시스템이 미종료 focus block 상태를 확인한다
  2. 사용자가 계속 / 종료 처리 중 하나를 선택한다
  3. 계속이면 세션을 복구한다
  4. 종료면 interrupted outcome을 기록한다
- 예외:
  - 세션 시간이 비정상적으로 오래 지나면 자동 종료 후보로 제안한다
- 결과:
  - 포커스 세션이 유실되지 않는다

### UC-45 quiet hours가 자정을 넘기는 사용자 설정

- 행위자: 사용자, 시스템
- 트리거: quiet hours 저장 또는 개입 평가
- 선행조건: quiet hours 사용
- 메인 흐름:
  1. 사용자가 예: `23:30 ~ 07:30` 같은 시간을 설정한다
  2. 시스템이 cross-midnight 구간으로 해석한다
  3. 개입 평가 시 해당 구간 전체를 억제 시간으로 본다
- 예외:
  - 안전 대응 개입은 예외 규칙을 따를 수 있다
- 결과:
  - 자정을 넘는 quiet hours가 안정적으로 동작한다

### UC-46 시간대 변경 후 집계 재해석

- 행위자: 시스템
- 트리거: 기기 timezone 변경 감지
- 선행조건: 기존 기록 존재
- 메인 흐름:
  1. 시스템이 timezone 변경을 감지한다
  2. 과거 이벤트는 당시 timezone 기준 local day를 유지한다
  3. 오늘/이번 주 계산만 현재 timezone 기준으로 다시 계산한다
- 예외:
  - 수면 기록의 대표 날짜는 여전히 wakeTime 기준을 유지한다
- 결과:
  - 여행이나 timezone 변경 후에도 집계가 크게 뒤틀리지 않는다

### UC-47 개입 만료 처리

- 행위자: 시스템
- 트리거: 일정 시간 지난 old intervention 존재
- 선행조건: 노출됐지만 반응 없는 개입
- 메인 흐름:
  1. 시스템이 개입의 유효 시간을 확인한다
  2. 현재 상태와 맞지 않게 된 개입은 만료 처리한다
  3. 필요하면 새로운 상태 기준으로 새 개입을 평가한다
- 예외:
  - 사용자가 이미 같은 목적 행동을 수동으로 완료했으면 즉시 만료한다
- 결과:
  - 오래된 제안이 계속 화면에 남지 않는다

### UC-48 기록 삭제 후 관련 이벤트 정리

- 행위자: 시스템
- 트리거: 체크인 또는 수면 기록 삭제
- 선행조건: 해당 기록과 연결된 이벤트 존재 가능
- 메인 흐름:
  1. 시스템이 `sourceRecordId` 연결 이벤트를 찾는다
  2. 집계 대상에서 제외한다
  3. 필요 시 append 정정 이벤트를 남긴다
- 예외:
  - 원본이 없어도 감사 목적 로그는 내부 보존 정책에 따라 남길 수 있다
- 결과:
  - 삭제 후 orphan 집계가 남지 않는다

### UC-49 설정 변경 즉시 반영

- 행위자: 사용자
- 트리거: 알림 빈도, quiet hours, 개입 강도 변경
- 선행조건: 설정 화면 진입
- 메인 흐름:
  1. 사용자가 설정을 바꾼다
  2. 시스템이 즉시 메모리 상태와 저장 상태를 업데이트한다
  3. 이후 개입 평가부터 새 규칙을 적용한다
- 예외:
  - 시스템 권한과 충돌하면 설정 설명을 함께 보여준다
- 결과:
  - 설정이 다음 앱 실행까지 기다리지 않고 즉시 적용된다

### UC-50 당일 `오늘은 그만` 이후 수동 복귀

- 행위자: 사용자
- 트리거: 사용자가 earlier suppression 후 스스로 앱 재진입
- 선행조건: 당일 `오늘은 그만` 상태 존재
- 메인 흐름:
  1. 시스템은 proactive 개입은 계속 억제한다
  2. 하지만 홈과 타임라인의 수동 진입은 막지 않는다
  3. 사용자가 원하면 체크인/회복 로그/집중 블록을 수동 시작할 수 있다
- 예외:
  - safety 대응은 suppression보다 우선할 수 있다
- 결과:
  - 사용자는 방해는 줄이되 자발적 사용은 계속할 수 있다

### UC-51 빈 타임라인에서 첫 행동 유도

- 행위자: 사용자
- 트리거: 타임라인 진입 시 기록 없음
- 선행조건: condition/sleep 기록 없음
- 메인 흐름:
  1. 시스템이 empty 상태를 보여준다
  2. `오늘 체크인` 또는 `회복 로그 남기기` CTA를 제공한다
  3. 사용자가 해당 플로우로 이동한다
- 예외:
  - 온보딩 미완료면 온보딩 재개가 우선한다
- 결과:
  - 빈 화면이 막다른 길이 되지 않는다

### UC-52 낮은 확신도 인사이트에서 추가 입력 유도

- 행위자: 시스템, 사용자
- 트리거: low-confidence insight 상태
- 선행조건: 데이터 부족
- 메인 흐름:
  1. 시스템이 강한 패턴 문장을 피한다
  2. 추가 체크인 또는 회복 로그 CTA를 노출한다
  3. 사용자가 보강 입력을 남긴다
  4. 이후 인사이트 신뢰도를 다시 계산한다
- 예외:
  - 사용자가 입력하지 않아도 low-confidence 상태를 유지한다
- 결과:
  - 확신 부족 상태가 다음 데이터 수집 행동으로 연결된다

### UC-53 홈/타임라인 부분 로딩

- 행위자: 시스템
- 트리거: 일부 데이터 소스만 로드 성공
- 선행조건: condition 또는 sleep 한쪽만 조회 가능
- 메인 흐름:
  1. 시스템이 가능한 데이터부터 화면을 렌더링한다
  2. 부족한 부분은 partial 상태로 표시한다
  3. CTA는 남은 부족 데이터를 보완하는 방향으로 계산한다
- 예외:
  - 전체 실패면 error 상태와 재시도를 보여준다
- 결과:
  - 한 소스 실패가 전체 화면 무력화로 이어지지 않는다

### UC-54 stale 알림 탭 처리

- 행위자: 사용자, 시스템
- 트리거: 이미 오래된 알림 탭
- 선행조건: 알림 발송 후 상태가 변했을 가능성 있음
- 메인 흐름:
  1. 사용자가 오래된 알림을 탭한다
  2. 시스템이 해당 개입이 아직 유효한지 재평가한다
  3. 유효하면 실행 플로우로 이동한다
  4. 무효면 홈 또는 현재 가장 적절한 화면으로 보낸다
- 예외:
  - 완료된 개입이면 기록 화면 또는 홈 요약으로 전환한다
- 결과:
  - stale 알림이 잘못된 행동 유도로 이어지지 않는다

## 12. 비즈니스 로직 유즈케이스

이 섹션은 화면 이동이 아니라 `시스템이 어떤 규칙으로 판단/계산/저장해야 하는가`를 유즈케이스로 정리한다.

### UC-55 홈 상태 입력 셋 구성

- 행위자: 시스템
- 트리거: 홈 모델 계산 시작
- 선행조건: 홈 진입 또는 refresh 필요
- 메인 흐름:
  1. 시스템이 최신 `ConditionRecord`를 조회한다
  2. 최근 7일 `ConditionRecord` 목록을 조회한다
  3. 오늘 `SleepRecord`를 조회한다
  4. 이 셋을 홈 상태 계산 입력으로 확정한다
- 예외:
  - 일부 소스가 없으면 partial 상태로 진행한다
- 결과:
  - 홈 계산 입력셋이 고정된다

### UC-56 condition score 계산

- 행위자: 시스템
- 트리거: 체크인 저장 또는 홈 상태 계산
- 선행조건: 필요한 점수값 존재
- 메인 흐름:
  1. 시스템이 energy, focus, recovery, stress 값을 읽는다
  2. 기본 공식을 적용한다
  3. score를 표시/집계용 값으로 만든다
- 예외:
  - 일부 선택값이 없으면 기본값 규칙을 적용한다
- 결과:
  - condition score가 일관되게 계산된다

### UC-57 attention flag 계산

- 행위자: 시스템
- 트리거: 최근 condition 기록 집계
- 선행조건: 최근 7일 기록 존재 가능
- 메인 흐름:
  1. 시스템이 각 condition record에 attention 조건을 적용한다
  2. `score < 55`, `stress >= 8`, `recovery <= 4` 여부를 확인한다
  3. 최근 7일 attention count를 계산한다
- 예외:
  - 불완전 기록은 집계 제외 규칙을 따른다
- 결과:
  - 주의 필요 건수가 계산된다

### UC-58 홈 CTA 우선순위 계산

- 행위자: 시스템
- 트리거: 홈 CTA 계산
- 선행조건: 홈 상태 계산 완료
- 메인 흐름:
  1. 오늘 체크인 없음 여부를 본다
  2. recovery 낮음 여부를 본다
  3. stress 높음 / attention flag 여부를 본다
  4. focus 낮음 여부를 본다
  5. sleep 없음 또는 평균 수면 짧음 여부를 본다
  6. 가장 높은 우선순위 1개 CTA를 선택한다
- 예외:
  - 동점 상황이어도 메인 CTA는 1개만 남긴다
- 결과:
  - 홈 메인 CTA가 결정된다

### UC-59 체크인 payload 정규화

- 행위자: 시스템
- 트리거: 체크인 저장 직전
- 선행조건: 사용자 입력 존재
- 메인 흐름:
  1. 시스템이 점수를 허용 범위로 clamp 한다
  2. 태그 중복을 제거한다
  3. 빈 메모를 null로 바꾼다
  4. `ConditionRecord` 저장 형태로 매핑한다
- 예외:
  - 필수값 누락 시 저장을 거부한다
- 결과:
  - 저장 가능한 정규화 payload가 생성된다

### UC-60 수면 payload 정규화

- 행위자: 시스템
- 트리거: 수면 저장 직전
- 선행조건: 취침/기상 입력 존재
- 메인 흐름:
  1. 시스템이 cross-day 시간 여부를 해석한다
  2. duration, latency, efficiency를 계산한다
  3. 음수 또는 초과 값을 clamp 한다
  4. `SleepRecord` 저장 형태로 매핑한다
- 예외:
  - 필수값 누락 시 저장을 거부한다
- 결과:
  - 저장 가능한 수면 payload가 생성된다

### UC-61 fallback descriptor 생성

- 행위자: 시스템
- 트리거: 체크인 저장 시 descriptor 미선택
- 선행조건: 태그 비어 있음
- 메인 흐름:
  1. 시스템이 energy/recovery/focus/stress 조합을 읽는다
  2. `Drained`, `Wired`, `Foggy`, `LockedIn`, `Overloaded`, `Steady` 중 하나를 고른다
- 예외:
  - 분기 충돌 시 우선순위 규칙을 적용한다
- 결과:
  - 대표 descriptor가 자동 생성된다

### UC-62 타임라인 엔트리 래핑

- 행위자: 시스템
- 트리거: 타임라인 날짜 조회 또는 월 조회
- 선행조건: `ConditionRecord` 또는 `SleepRecord` 존재
- 메인 흐름:
  1. 시스템이 각 기록을 내부 단일 엔트리 포맷으로 변환한다
  2. condition와 sleep 엔트리를 같은 시간 축에 정렬한다
  3. UI에 동일 구조로 전달한다
- 예외:
  - 의미 복원이 불가능한 레거시 엔트리는 제외한다
- 결과:
  - 타임라인이 하나의 흐름으로 렌더링된다

### UC-63 주간 요약 계산

- 행위자: 시스템
- 트리거: 주간 요약 카드 생성
- 선행조건: 최근 7일 데이터 조회 가능
- 메인 흐름:
  1. 시스템이 최근 7일 condition/sleep 데이터를 모은다
  2. 평균 score, energy, focus, recovery, stress, sleep hours, efficiency를 계산한다
  3. top signals와 stable count를 계산한다
- 예외:
  - 데이터 부족 시 low-confidence 상태로 표시한다
- 결과:
  - 주간 요약 입력값이 생성된다

### UC-64 인사이트 문장 선택

- 행위자: 시스템
- 트리거: 인사이트 생성
- 선행조건: 주간 요약 계산 완료
- 메인 흐름:
  1. 시스템이 데이터 부족 여부를 확인한다
  2. score 높음/수면 충분, stress 높고 recovery 낮음, energy 높고 focus 낮음 등 우선순위를 적용한다
  3. 가장 맞는 인사이트 문장 패턴 1개를 선택한다
- 예외:
  - low-confidence면 강한 문장을 쓰지 않는다
- 결과:
  - 경향 기반 인사이트가 생성된다

### UC-65 contextual sampling 판단

- 행위자: 시스템
- 트리거: 질문 또는 micro check-in 필요성 평가
- 선행조건: 상태 불확실 또는 개입 의사결정 시점
- 메인 흐름:
  1. 시스템이 불확실성, 행동 도움성, 응답 가능성, 최근 중복 질문 여부를 평가한다
  2. 질문할지 침묵할지 결정한다
- 예외:
  - active focus block, 바쁜 캘린더 구간, quiet hours면 억제 우선
- 결과:
  - 질문 노출 여부가 결정된다

### UC-66 intervention suppression 판단

- 행위자: 시스템
- 트리거: proactive 개입 평가
- 선행조건: 개입 후보 존재
- 메인 흐름:
  1. 시스템이 quiet hours, dismiss 누적, focus block, stronger surface 중복 여부를 확인한다
  2. suppression 적용 여부를 결정한다
- 예외:
  - safety 대응이면 suppression보다 안전 우선순위를 적용한다
- 결과:
  - 개입 억제 또는 허용이 결정된다

### UC-67 intervention surface 선택

- 행위자: 시스템
- 트리거: 개입 노출 허용
- 선행조건: 개입 필요 + suppression 없음
- 메인 흐름:
  1. 시스템이 사용자가 앱 안에 있는지 확인한다
  2. 홈 CTA, 인앱, 시스템 알림 우선순위를 적용한다
  3. 같은 세션 중복 표면 노출을 막는다
- 예외:
  - weaker surface로 충분하면 stronger surface를 쓰지 않는다
- 결과:
  - 개입 표면이 결정된다

### UC-68 outcome state 판정

- 행위자: 시스템
- 트리거: 개입 반응 또는 실행 종료
- 선행조건: 노출된 개입 존재
- 메인 흐름:
  1. 시스템이 shown/opened/accepted/completed/dismissed/ignored/snoozed 중 상태를 고른다
  2. 상태 변화가 있으면 improved/unchanged/worsened/unknown을 함께 기록한다
- 예외:

## 13. 현재 제품에 남기는 추가 비즈니스 로직 유즈케이스

이 섹션에는 현재 FullCon 메인 제품에 실제로 남기는 내부 로직만 둔다.

제거 원칙:

- `garden`, `plant assignment`, `quest`, `level`, `growth`는 현재 제품 유즈케이스에 포함하지 않는다
- 독립 기능으로서의 `caring`, `anger`는 유지하지 않는다
- 남는 것은 `현재 코어 루프를 직접 지지하는 로직`과 `필수 호환 로직`뿐이다

- 트리거: 홈 refresh

  2. attention 조건에 걸리는 레코드를 추린다
  3. 이를 홈의 attention 대상 목록으로 유지한다
- 예외:
  - 기록이 없으면 빈 목록으로 둔다

  - 홈은 주의 필요 건수와 대상 목록을 가진다

### UC-77 체크인 파생 affect 신호 계산

  - 푸시 실패가 개입 유실로 바로 이어지지 않는다
