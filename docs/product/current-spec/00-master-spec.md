# FullCon Canonical Spec

작성일: 2026-04-30

## 0. 문서 원칙

이 문서는 현재 `FullCon`의 단일 정본 스펙이다.

원칙:

- 제품 정의
- 시스템 구조
- 화면 계약
- 데이터 계약
- 구현 원칙

위 다섯 축은 이 문서 하나를 기준으로 해석한다.

기존 `10-screen-contracts.md`, `20-data-logic-contracts.md`, `30-vibe-coding-playbook.md`의 내용은 이 문서로 병합되었다.

### 0.1 해석 우선순위

이 문서를 읽을 때는 아래 순서로 해석한다.

1. 사용자에게 실제로 보여야 하는 화면/행동 결과
2. 저장되어야 하는 데이터와 이벤트
3. 그 결과를 만들기 위한 로직 규칙
4. 그 뒤에 제품 의도와 철학

즉, 추상 원칙보다 `앱에서 반드시 구현되어야 하는 동작`이 우선한다.

## 1. 제품의 최종 정의

FullCon은 `컨디션 기록 앱`이 아니다.

현재 제품 정의는 아래 한 문장으로 고정한다.

> FullCon은 사용자의 상태, 행동, 맥락을 지속적으로 학습하고 필요한 순간에만 개입하여 하루 퍼포먼스를 유지시키는 개인화된 AI 코치 시스템이다.

핵심은 `기록`이 아니라 `개입`이고, `대시보드`가 아니라 `운영`이다.

### 1.1 현재 목표

현재 단계의 제품 목표는 아래 4개다.

1. 사용자가 빠르게 현재 상태를 남길 수 있어야 한다
2. 앱이 지금 필요한 행동 1개를 명확히 제안해야 한다
3. 개입 이후 결과를 추적해 다음 판단에 반영할 수 있어야 한다
4. 외부 데이터가 없어도 핵심 루프가 동작해야 한다

### 1.2 현재 비목표

현재 단계에서 아래는 메인 목표가 아니다.

- 장문의 AI 상담 대화
- 과도한 리포트 대시보드
- 감성 세계관 복원
- 소셜 기능
- 복잡한 게임화
- 외부 데이터 없이는 쓸 수 없는 구조

### 1.3 핵심 용어

- `컨디션`: 사용자의 현재 상태를 해석한 결과
- `회복`: 수면과 휴식으로 대표되는 복구 상태
- `체크인`: 사용자가 직접 남기는 상태 입력
- `개입`: 시스템이 먼저 제안하거나 묻는 행동
- `표면(surface)`: 홈, 인앱, 알림처럼 개입이 노출되는 위치
- `결과(outcome)`: 개입 이후 실제로 벌어진 행동과 상태 변화
- `개인화`: 반복된 행동/결과를 바탕으로 규칙이 사용자별로 달라지는 것

### 1.4 안전 경계

FullCon은 의료 진단 시스템이 아니다.

원칙:

- 질병, 장애, 응급 상태를 확정적으로 판단하지 않는다
- 일반 코칭 개입과 안전 대응을 혼동하지 않는다
- 위험 신호가 의심될 때는 코칭 품질보다 안전한 축소 응답이 우선한다
- 의학적 조언을 제공하는 것처럼 보이는 카피를 쓰지 않는다

### 1.5 현재 릴리스 구현 스펙

이 섹션은 현재 앱에서 `실제로 구현되어야 하는 결과물`만 요약한다.

#### 1.5.1 앱 시작과 진입

- 앱은 `/dev`가 아니라 `/` 또는 `/onboarding`에서 시작해야 한다
- 첫 실행 사용자는 온보딩으로 간다
- 온보딩 완료 사용자는 홈으로 간다
- 온보딩 미완료 상태에서 메인 라우트 딥링크가 들어오면 온보딩을 먼저 재개한다

#### 1.5.2 온보딩 결과

- 사용자는 닉네임을 입력할 수 있어야 한다
- 사용자는 첫 컨디션 체크인을 완료할 수 있어야 한다
- 첫 체크인 저장 이후에만 온보딩 완료가 된다
- 중간 종료 후 재진입 시 마지막 안전 단계부터 복구 가능해야 한다

#### 1.5.3 홈 결과

홈 첫 화면에는 아래 3개만 핵심 블록으로 보여야 한다.

1. `오늘 상태`
2. `지금 할 일`
3. `이번 주 한 줄`

추가 규칙:

- 메인 CTA는 1개만 보여야 한다
- CTA는 현재 상태 기준 가장 필요한 행동으로 연결되어야 한다
- 오늘 체크인이 없으면 체크인 CTA가 우선이다
- 회복 데이터가 없거나 낮으면 회복 로그 CTA가 우선이다

#### 1.5.4 컨디션 체크인 결과

- 사용자는 30초 안에 기본 체크인을 저장할 수 있어야 한다
- 최소 입력은 `에너지`, `회복`이다
- 선택 입력으로 `집중`, `스트레스`, `태그`, `상황`, `몸 반응`, `메모`를 남길 수 있다
- 저장 후 홈 데이터가 즉시 갱신되어야 한다
- 저장 자체가 심리 평가처럼 느껴지면 안 된다

#### 1.5.5 회복 로그 결과

- 사용자는 취침/기상 시간과 회복감을 저장할 수 있어야 한다
- 기존 수면 기록을 수정할 수 있어야 한다
- 저장 후 현재 화면이 닫히고 홈/타임라인에서 반영되어야 한다
- 수면 계산은 저장 시 정규화되어야 한다

#### 1.5.6 타임라인 결과

- 사용자는 월간 캘린더에서 기록이 있는 날짜를 볼 수 있어야 한다
- 특정 날짜를 누르면 condition + sleep 기록이 한 흐름으로 보여야 한다
- 주간 요약은 최근 7일 기준으로 계산되어야 한다
- 인사이트는 확정 진단이 아니라 경향 요약이어야 한다

#### 1.5.7 개입 시스템 결과

- 앱은 하루에 매우 제한된 수의 proactive 개입만 해야 한다
- 개입은 홈 CTA, 인앱 표면, 알림 중 하나로 노출된다
- 이미 앱 안에 있으면 알림보다 인앱 표면이 우선이다
- 사용자가 dismiss하거나 수동 행동을 하면 같은 목적 개입은 줄어들어야 한다
- quiet hours에는 일반 상태 질문을 보내지 않는다

#### 1.5.8 데이터와 학습 결과

- 체크인 저장 시 `ConditionRecord`가 저장되어야 한다
- 회복 로그 저장 시 `SleepRecord`가 저장되어야 한다
- 개입 노출/반응/완료/피드백은 이벤트로 남아야 한다
- 집계는 삭제, 중복, 미완료 draft를 제외하고 계산되어야 한다
- 외부 권한이 없어도 로컬 데이터만으로 핵심 루프가 동작해야 한다

#### 1.5.9 릴리스 차단 조건

아래 중 하나라도 맞으면 현재 릴리스 기준 미완성으로 본다.

- 앱이 `/dev`에서 시작한다
- 온보딩 완료 전 사용자가 홈 플로우에 정상 진입한다
- 홈에 메인 CTA가 2개 이상 뜬다
- 체크인/수면 저장 후 홈이나 타임라인이 갱신되지 않는다
- 알림 권한이 없을 때 핵심 루프가 막힌다
- 집계가 삭제/중복 데이터까지 포함해 왜곡된다

#### 1.5.10 추가로 필요한 기능 묶음

현재 스펙을 실제 제품으로 닫기 위해 아래 기능이 추가로 필요하다.

Core Loop 필수:

- 개입 실행 결과 기록 기능
- 개입 dismiss / snooze / 완료 분기 처리
- 기록 수정 / 삭제 / 복구 후 집계 재계산
- 미체크인 / 미회복 상태에 대한 복귀 유도 흐름

개입 정확도 필수:

- 상태 불확실 시 micro check-in 호출
- 취침 전 wind-down 개입
- 집중 블록 타이머와 중간 확인
- stale data 감지와 데이터 신선도 경고

사용자 통제 필수:

- 알림 on/off와 quiet hours 설정
- 개입 빈도 조절
- 개입 미루기 / 오늘 그만 보기
- 권한 거부 후 재설정 진입

운영 안정화 필수:

- 이벤트 저장 실패 재시도
- 홈/타임라인 강제 refresh
- 집계 제외 데이터 디버그 확인
- 마이그레이션 후 데이터 호환 확인

## 2. 제품 방향 전환

기존 사고:

- 사용자가 앱을 연다
- 상태를 기록한다
- 나중에 돌아본다

현재 방향:

- 앱이 먼저 상황을 이해한다
- 필요한 순간에만 묻는다
- 바로 행동을 제안한다
- 결과를 학습한다

제품 진화의 한 줄은 아래와 같다.

> Passive -> Proactive -> Predictive

## 3. 시스템 본질

FullCon은 아래 루프를 반복하는 `Closed Loop` 시스템이다.

1. 데이터 수집
2. 상태 추정
3. 미래 예측
4. 개입 결정
5. 행동 실행
6. 결과 수집
7. 피드백 반영
8. 학습
9. 반복

화면은 이 루프를 돕는 인터페이스다. 제품의 본체는 루프다.

## 4. 핵심 엔진 구조

### 4.1 Data Engine

주요 입력:

- 체크인
- 수면
- 폰 사용
- 워치
- 캘린더
- 행동 로그

원칙:

- 데이터는 많이 모으는 것이 목적이 아니다
- 개입 타이밍과 행동 제안 정확도를 높이기 위해 수집한다
- AI보다 데이터 구조가 먼저다

### 4.2 Contextual Sampling Engine

질문을 `언제` 던질지 결정한다.

원칙:

- 랜덤 질문 금지
- 상황 기반 질문 우선
- 사용자가 바쁘거나 몰입 중이면 침묵할 수 있어야 한다

### 4.3 Micro Check-in Engine

1초 안에 응답 가능한 최소 입력 계층이다.

예시 UX:

- `집중 잘됨`
- `보통`
- `집중 안됨`

원칙:

- 길게 쓰게 하지 않는다
- 감정 서술보다 상태 판별을 우선한다
- 질문 횟수보다 응답 마찰 최소화가 중요하다

### 4.4 Prediction Engine

현재 상태를 해석하는 것을 넘어 `앞으로 무너질 가능성`을 예측한다.

예시:

- 집중 붕괴 위험 시간대
- 회복 부족으로 인한 저하 구간
- 취침 지연 가능성

### 4.5 Intervention Engine

언제, 무엇을, 어떤 방식으로 개입할지 결정한다.

개입 예시:

- 취침 유도
- 회복 유도
- 집중 시작 제안
- 짧은 상태 확인 질문

핵심은 `개입 수`가 아니라 `개입 정확도`다.

### 4.6 Personalization Engine

행동과 결과를 연결해 사용자별 패턴을 학습한다.

핵심 질문:

- 이 사용자는 언제 무너지는가
- 어떤 제안에 실제로 반응하는가
- 어떤 시간대와 맥락에서 개입 성공률이 높은가

### 4.7 State Estimation Layer

각 입력을 그대로 노출하지 않고 `지금 사용자에게 무슨 상태가 벌어지고 있는지`를 추정하는 중간 계층이다.

현재 기준 핵심 상태 축:

- 에너지
- 집중
- 회복
- 스트레스
- 안정성

원칙:

- 원시 입력보다 해석된 상태를 우선 사용한다
- 화면과 개입 로직은 같은 상태 해석을 공유해야 한다
- UI와 알림이 서로 다른 판정을 내리면 안 된다

### 4.8 Outcome & Learning Layer

개입 이후 무엇이 실제로 달라졌는지 평가하는 계층이다.

추적 대상:

- 개입 노출
- 개입 수락 여부
- 행동 완료 여부
- 상태 개선 여부
- 사용자 명시 피드백

원칙:

- 제안 자체보다 결과를 더 중요하게 기록한다
- `보였다`와 `실행했다`를 구분한다
- 성공/실패 데이터가 다음 개입 정책을 바꿔야 한다

## 5. 전역 UX 원칙

### 5.1 Passive

앱을 열지 않아도 시스템이 동작해야 한다.

- 상태 감지
- 타이밍 판단
- 알림 또는 개입

### 5.2 Active

앱을 열었을 때는 현재 판단과 다음 행동이 바로 보여야 한다.

- 오늘 상태
- 지금 할 일 1개

### 5.3 Reflective

축적된 행동과 결과에서 패턴과 인사이트를 읽게 한다.

- 반복되는 저하 패턴
- 회복과 퍼포먼스의 연결
- 개입 성공/실패 흐름

### 5.4 행동 중심

- 읽기용 설명보다 실행형 CTA를 우선한다
- 한 화면의 첫 시선에서 판단할 블록은 최소화한다
- 화면당 메인 액션은 가능하면 1개만 둔다

### 5.5 최소 개입

- 하루 1~2회 수준의 적은 개입이 기본값이다
- 질문은 점점 줄어들어야 한다
- 원시 데이터는 숨기고 해석만 보여준다

### 5.6 신뢰와 통제감

- 사용자는 감시당한다고 느끼면 안 된다
- 언제든 입력을 건너뛸 수 있어야 한다
- 개입은 명령이 아니라 코치의 제안처럼 느껴져야 한다
- 데이터 수집 목적은 반드시 행동 개선과 연결되어야 한다

### 5.7 자기결정감 중심

심리적으로 지속 가능한 제품은 `통제`보다 `자기결정감`을 강화해야 한다.

원칙:

- 사용자가 선택했다고 느껴야 행동 유지가 일어난다
- 시스템은 강요보다 선택지를 좁혀 주는 역할을 한다
- 개입은 `해야 한다`보다 `지금 이게 도움이 될 수 있다` 톤을 우선한다
- 사용자의 자율성, 유능감, 회복 가능성을 깎아먹는 피드백을 금지한다

### 5.8 실패 정상화

행동 변화는 선형이 아니라 들쭉날쭉한 과정으로 본다.

원칙:

- 체크인 누락, 수면 실패, 개입 무시는 `실패한 사용자`의 증거가 아니다
- lapse는 다음 설계 개선의 신호로 해석한다
- `망쳤다`, `또 실패했다` 같은 자기비난 프레이밍을 만들지 않는다
- 복귀 장벽은 항상 낮게 유지한다

## 6. 정보 구조와 라우트

메인 라우트:

| 경로 | 이름 | 역할 |
| --- | --- | --- |
| `/onboarding` | 온보딩 | 첫 사용자 진입 |
| `/` | 홈 | 오늘 상태 + 지금 할 일 + 주간 한 줄 |
| `/seeding` | 컨디션 체크인 | 상태 입력 |
| `/sleep` | 회복 로그 | 수면/기상 상태 입력 |
| `/history` | 타임라인 | 월간 흐름 + 날짜별 기록 |
| `/settings` | 설정 | 테마/언어/앱 정보/세션 |

레거시 또는 내부 도구:

- `/dev`
- `/login`

제거 또는 비활성화 대상 레거시 개념:

- garden
- plant assignment 노출
- quest / level / growth
- caring 모듈을 독립 제품 기능으로 보는 방식
- anger 모듈을 독립 제품 기능으로 보는 방식

메인 스펙은 위 레거시 영역과 개념을 핵심 기능으로 취급하지 않는다.

### 6.1 Runtime Entry & Route Guards

정상 사용자 진입 규칙:

- 앱의 기본 시작점은 항상 `/` 또는 `/onboarding`이다
- 앱이 `/dev`에서 시작하면 안 된다
- 온보딩 미완료 사용자는 우선 `/onboarding`을 거친다
- 온보딩 완료 후에는 원래 가려던 라우트가 있으면 그곳으로, 없으면 `/`로 보낸다

릴리스 기준 가드 규칙:

- `/onboarding`, `/`, `/seeding`, `/sleep`, `/history`, `/settings`는 메인 제품 라우트다
- 온보딩 미완료 상태에서 메인 라우트 딥링크가 들어오면 먼저 온보딩을 수행한다
- `/dev`와 `/login`은 내부/레거시 경로로 보고 일반 사용자 플로우의 일부로 취급하지 않는다

개발 중 예외 규칙:

- 디버그 편의를 위한 직접 진입은 허용할 수 있다
- 다만 이 예외는 릴리스 동작의 정본이 아니다
- 디버그 예외를 이유로 제품 가드를 약화시키지 않는다

### 6.2 Permission & Degradation Strategy

권한 의존 기능은 핵심 루프 위에 올라가야 한다.

원칙:

- 체크인, 홈, 회복 로그, 타임라인은 권한 없이도 동작해야 한다
- 알림 권한이 없으면 인앱 표면만 사용한다
- 캘린더/워치/폰 사용 권한이 없으면 해당 신호만 빠진 상태로 판정한다
- 권한 거부를 실패로 취급하지 않고 graceful degradation으로 처리한다

소스별 폴백:

- 알림 없음: 홈 CTA + 인앱 표면만 사용
- 캘린더 없음: 바쁨/비어 있음 같은 일정 맥락 추정을 비활성화
- 워치 없음: 생리 상태 축은 미지원으로 두고 체크인/수면만 사용
- 폰 사용 없음: usage 기반 붕괴 탐지는 끄고 self-report를 우선한다

UI 규칙:

- 권한이 없어도 메인 라우트 접근은 막지 않는다
- 권한이 필요한 확장 기능만 축소한다
- 권한 거부 상태에서는 `허용해야 사용 가능`이 아니라 `허용하면 더 정확해짐` 톤을 쓴다

금지:

- 권한이 없으면 앱의 핵심 루프가 막히는 구조
- 권한 요청 이유 없이 선제적으로 많은 권한을 묶어 요청하는 것

### 6.3 User Identity & Session Semantics

현재 FullCon은 `단일 로컬 사용자`를 기본 가정으로 한다.

규칙:

- 하나의 로컬 저장소는 한 사용자의 기록으로 해석한다
- `로그아웃`은 계정 전환이 아니라 세션 초기화 수준으로 본다
- 세션 초기화만으로 로컬 기록이 자동 삭제되지는 않는다
- 멀티 계정, 멀티 프로필, 공유 기기 사용자 전환은 현재 스코프 밖이다

### 6.4 Partial Onboarding Resume

온보딩은 중간 중단 후 복귀 가능해야 한다.

규칙:

- 닉네임 입력 후 종료되면 다음 진입 시 해당 단계 이후부터 복구할 수 있어야 한다
- 첫 체크인 저장 전에는 `isCompleted`를 `true`로 바꾸지 않는다
- 첫 체크인 작성 중 중단되면 draft 복구 또는 안전한 재시작 중 하나를 일관되게 택해야 한다
- 온보딩 미완료 상태에서는 홈 대신 온보딩 재개가 우선한다

## 7. 주요 기능 정의

### 7.1 실행형 제안

예시:

- `20분 회복 시작`
- `25분 집중 시작`

### 7.2 타이밍 개입

우선 감지 대상:

- 취침이 필요한 시점
- 집중이 무너지는 시점
- 회복이 필요한 시점

### 7.3 상황 기반 질문

질문은 기록 수집용이 아니라 상태 판별용이다.

예시:

- `지금 상태를 잠깐 알려주세요`

### 7.4 집중 블록 코칭

예시 흐름:

1. `2시간 집중할게요`
2. 중간 상태 확인
3. 종료 후 결과 피드백

### 7.5 개인화 학습

반복 관찰을 통해 아래를 점점 더 정확히 맞혀야 한다.

- 언제 개입해야 하는지
- 어떤 제안을 해야 하는지
- 어떤 표현 방식이 가장 잘 먹히는지

### 7.6 Outcome Tracking

모든 핵심 행동은 결과 추적 가능해야 한다.

최소 추적 대상:

- 체크인 완료
- 회복 로그 완료
- 제안 CTA 클릭
- 집중 블록 시작 / 종료
- 개입 무시 / 닫기 / 미루기

목적:

- 행동이 실제로 일어났는지 안다
- 개입의 품질을 사후 평가한다
- 개인화 근거를 쌓는다

### 7.7 Feedback System

결과만으로 해석이 어려운 경우 짧은 피드백을 받는다.

예시:

- `도움 됐어요`
- `그냥 넘길게요`
- `지금은 타이밍이 아니에요`

원칙:

- 장문 회고를 강요하지 않는다
- 개입 직후 또는 세션 종료 직후에만 묻는다
- 피드백이 없더라도 시스템은 기본 outcome을 기록해야 한다

### 7.8 Notification & Surface Strategy

개입 표면은 하나로 고정하지 않는다.

우선 표면:

- 홈의 메인 CTA
- 인앱 배너 또는 카드
- 시스템 알림

원칙:

- 같은 메시지를 여러 표면에 중복 노출하지 않는다
- 가장 약한 표면으로 해결 가능하면 알림까지 올리지 않는다
- 시스템 알림은 정말 타이밍 가치가 있을 때만 사용한다

### 7.9 Reflective Insight Principles

회고와 인사이트는 `진단`이 아니라 `가설`이어야 한다.

원칙:

- 인사이트는 확정 문장보다 탐색 문장으로 제시한다
- 사용자가 스스로 패턴을 검토할 여지를 남긴다
- 정서나 성격을 본질화하는 라벨을 붙이지 않는다
- 가치, 동기, 회복 패턴 해석은 행동 지원에 도움이 될 때만 노출한다

### 7.10 Reinforcement Principles

행동 변화는 보상보다 `즉시성`, `작은 성공`, `부담 감소`에 더 민감하게 설계한다.

원칙:

- 큰 보상보다 작은 성공 경험을 빠르게 느끼게 한다
- 체크인 자체보다 체크인 이후 행동 변화가 강화 대상이다
- streak 집착이나 결손 공포를 유발하는 강화는 피한다
- 강화는 죄책감이 아니라 효능감으로 연결되어야 한다

### 7.11 Intervention Execution Flow

개입은 노출만으로 끝나면 안 되고 실제 실행 흐름을 가져야 한다.

최소 요구:

- CTA 클릭 후 실행 상태로 진입
- 실행 중 상태 표시
- 완료 / 중단 / 미루기 분기
- 종료 후 outcome 기록

예시:

- `20분 회복 시작` -> 타이머 또는 실행 화면 -> 완료/중단 -> 피드백
- `25분 집중 시작` -> focus block 시작 -> 중간 확인 -> 종료 피드백

### 7.12 Snooze & Suppression Controls

사용자는 개입을 완전히 거절하지 않고도 `나중에`로 보낼 수 있어야 한다.

필수 동작:

- `10분 후`
- `오늘 저녁`
- `오늘은 그만`

효과:

- snooze는 같은 목적 개입의 즉시 재노출을 막는다
- `오늘은 그만`은 당일 proactive budget을 강하게 줄인다

### 7.13 Bedtime Wind-down Feature

취침 개입은 단순 알림이 아니라 짧은 취침 유도 흐름이어야 한다.

필수 요소:

- 취침 전 개입 트리거
- 지금 자도 되는지 / 미루는지 선택
- 짧은 준비 행동 제안
- 실제 취침 기록으로 연결

예시 행동:

- `15분 후 눕기`
- `지금 화면 끄기`
- `취침 기록 남기기`

### 7.14 Data Freshness Prompt

시스템은 데이터가 오래되었을 때 그것을 `정확도 저하`로 알려야 한다.

필수 동작:

- 오늘 체크인이 없음
- 최근 회복 로그 없음
- 최근 행동 데이터 부족

표면:

- 홈 배너
- 메인 CTA
- 필요 시 micro check-in

### 7.15 Micro Check-in Trigger Feature

micro check-in은 항상 열려 있는 화면이 아니라 특정 조건에서만 떠야 한다.

우선 트리거:

- 상태 불확실성이 높음
- 최근 데이터가 오래됨
- passive signal과 self-report가 충돌함
- focus block 중간 확인 시점

응답 형식:

- one-tap 또는 very short choice
- 3개 내외 옵션

### 7.16 Record Edit & Delete Feature

사용자는 자신의 기록을 사후 수정하거나 삭제할 수 있어야 한다.

필수 범위:

- 컨디션 체크인 수정
- 회복 로그 수정
- 기록 삭제

시스템 동작:

- 수정/삭제 후 홈, 타임라인, 집계, 인사이트가 다시 계산된다
- 삭제는 기록 숨김이 아니라 실제 집계 제외로 이어진다

### 7.17 Weekly Review Feature

주간 요약은 읽는 카드만이 아니라 짧은 회고 진입점이어야 한다.

필수 요소:

- 최근 7일 핵심 패턴 1~2개
- 가장 필요한 다음 행동 1개
- 사용자 확인 또는 공감 피드백 1탭

금지:

- 리포트 페이지처럼 지표만 나열하는 것

### 7.18 Notification Preferences Feature

사용자는 개입을 받을 방식과 강도를 조절할 수 있어야 한다.

최소 설정:

- 알림 on/off
- quiet hours
- 개입 빈도 낮춤
- focus block 중 방해 최소화

### 7.19 Permission Recovery Feature

권한을 거부한 사용자가 나중에 다시 설정할 수 있어야 한다.

필수 동작:

- 설정 화면에서 각 권한 상태 확인
- 왜 필요한지 짧은 설명
- 시스템 설정으로 이동
- 권한 없이도 가능한 대안 안내

### 7.20 Intervention History Feature

시스템은 최근 어떤 제안을 했고 사용자가 어떻게 반응했는지 내부적으로 추적할 수 있어야 한다.

목적:

- 중복 개입 방지
- dismiss 누적 반영
- 최근 completed 개입 재노출 방지

최소 기록:

- 개입 종류
- 노출 시각
- 표면
- 결과 상태

### 7.21 Confidence-aware Insight Feature

데이터가 부족한 상태에서는 강한 인사이트 대신 낮은 확신도 표현이 필요하다.

필수 동작:

- 데이터 부족 시 `아직 패턴을 보는 중` 상태 사용
- 충분한 데이터가 쌓였을 때만 강한 경향 문장 노출
- low-confidence 상태에서는 추가 입력 CTA를 우선한다

## 8. 화면 계약

### 8.1 온보딩

목적:

- 앱의 정체성을 `기록 앱`이 아니라 `컨디션 코치`로 인식시킨다
- 사용자가 닉네임을 입력하고 첫 체크인을 마치게 한다

단계 계약:

1. Intro

- FullCon 워드마크
- 한 줄 제품 정의
- `회복 / 체크인 / 제안` 3축 미리보기
- 시작 CTA

금지:

- 추상 연출만 있고 제품 설명이 없는 화면
- 긴 감성 카피
- 철학 설명 덤프

2. Identity

- 닉네임 입력
- 다음 단계로 가는 명확한 버튼

3. First Check-in

- 첫 체크인 입력 UI
- 저장 후 요약
- 홈 이동 CTA

완료 조건:

- 닉네임 저장
- 온보딩 완료 플래그 저장
- 홈 진입

### 8.2 홈

목적:

- 오늘 상태를 한 번에 읽게 한다
- 지금 해야 할 행동 1개를 정해준다
- 이번 주 흐름을 한 줄로 압축한다

첫 시선 블록은 아래 3개뿐이다.

1. `오늘 상태`
2. `지금 할 일`
3. `이번 주 한 줄`

`오늘 상태` 카드 필수 요소:

- 점수 링
- 상태 헤드라인 1개
- 제안 문장 1개
- 상태 요약 1줄

`지금 할 일` 규칙:

- 메인 CTA 1개만 노출
- 행동 제목
- 짧은 설명

`이번 주 한 줄` 규칙:

- 요약 제목
- 인사이트 본문
- 하이라이트 1~2개
- 타임라인 이동 링크

금지:

- CTA 2개 병렬 배치
- 홈에서 리포트 카드 과다 확장
- 상단 우측 설정 버튼

하단 네비 항목:

- 오늘
- 타임라인
- 회복
- 설정

### 8.3 컨디션 체크인

목적:

- 사용자가 30초 안에 오늘 상태를 남길 수 있어야 한다
- 최소 입력만으로 홈 제안이 달라지게 해야 한다

필수 요소:

- 제목: `오늘 상태` 또는 동등한 짧은 표현
- 에너지 슬라이더
- 회복 슬라이더
- 현재 점수 또는 대표 상태 표시
- 저장 버튼

선택 펼침 영역:

- 집중
- 스트레스
- 상태 태그
- 영향 준 상황
- 몸 반응
- 메모

상호작용 규칙:

- 첫 진입 시 상태 reset
- 기본 입력만으로 저장 가능
- 상세 입력은 기본적으로 접힘
- 저장 성공 시 홈으로 복귀하고 데이터 refresh
- 정교한 감정 라벨링을 매번 강요하지 않는다
- 사용자가 확신 없는 항목은 비워둘 수 있어야 한다

금지:

- 질문형 문장 과다 노출
- 감정 좌표/사분면 중심 구조
- 씨앗/물 주기 메타포
- 자기평가를 도덕 평가처럼 느끼게 하는 문구

### 8.4 회복 로그

목적:

- 수면과 기상 상태를 기록해 회복 흐름을 읽게 한다
- 홈의 제안 정확도를 높인다

필수 요소:

- 취침 시간
- 기상 시간
- 총 수면 시간
- 입면 잠복기
- 회복감
- 수면 질
- 태그
- 저장 CTA

선택 요소:

- 자연 기상
- 스누즈 여부
- 메모

규칙:

- 새 기록 생성 가능
- 기존 기록 수정 가능
- 저장 후 현재 화면 `pop`

금지:

- 과도한 과학 설명
- 홈보다 더 강한 대시보드 역할

### 8.5 타임라인

목적:

- 월 단위로 condition + recovery 흐름을 본다
- 특정 날짜의 기록을 자세히 확인한다
- 주간 요약을 함께 제공한다

구성:

- 상단: 뒤로가기, 화면 제목, 안내문 1줄, 월간 캘린더
- 하단 드래그 시트: 이번 주 요약 카드, 선택 날짜 엔트리 리스트

엔트리 타입:

- 컨디션 체크인
- 회복 로그

엔트리 카드 공통 요소:

- 제목
- 시각
- 상태/점수 보조칩
- 메모 또는 요약 1~2줄

금지:

- `EmotionRecord` 용어 전면 노출
- condition/sleep UI 체계 완전 분리
- `정원 일지` 같은 레거시 네이밍

빈 상태:

- `아직 기록이 없어요. 오늘 컨디션 체크인이나 회복 로그를 남겨보세요.`

원칙:

- 패턴 요약은 `가능성`, `경향`, `가설` 언어를 우선한다
- 사용자를 단일 성격 유형으로 고정하는 설명을 피한다

포함 요소:

- 버전

### 8.7 Dev / Legacy

- 새 기능은 이 레거시 개념을 기반으로 확장하지 않는다

### 8.8 Screen State Contracts

핵심 화면은 happy path만 정의해서는 안 된다.

홈:

- loading: skeleton 또는 간단한 placeholder를 보여준다
- empty: `오늘 상태`와 `첫 행동 CTA`를 우선 노출한다
- partial: condition만 있고 sleep이 없으면 회복 로그 CTA를 우선한다
- error: 짧은 에러 문장 + 재시도 액션 1개

컨디션 체크인:

- idle: 기본 입력만 바로 조작 가능해야 한다
- saving: 저장 중복 입력을 막고 진행 상태를 보여준다
- success: 홈 복귀와 refresh를 보장한다
- error: 저장 영역 가까이에 짧은 실패 메시지를 노출한다

회복 로그:

- create: 새 기록 생성 상태
- edit: 기존 값 hydrate 상태
- saving: 중복 저장 방지
- error: 입력값은 유지하고 실패 메시지만 보여준다

타임라인:

- loading: 월간 로딩 상태를 보여준다
- empty: 체크인/회복 로그 유도 문구를 노출한다
- populated: condition + sleep를 같은 흐름으로 보여준다
- error: 월 전환 또는 재시도 액션을 제공한다

설정:

- current value가 항상 먼저 보여야 한다
- 토글/선택 변경 실패 시 즉시 원복되거나 명확한 실패 상태를 보여준다

### 8.9 Permission Request States

권한 요청이 필요한 표면은 아래 상태를 구분한다.

- pre-permission: 왜 필요한지 짧게 설명
- system-prompt: OS 권한 요청
- granted: 기능 활성화
- denied: 대체 가능한 축소 흐름 제공
- permanently-denied: 설정 이동 안내와 비권한 대안 제공

원칙:

- 권한이 없더라도 사용자가 다음 행동을 계속할 수 있어야 한다
- 권한 요청 문구는 데이터 수집이 아니라 사용자 가치로 설명한다

### 9.5 Event & Learning Schema

Closed Loop를 구현하기 위한 정식 이벤트 계층을 둔다.

핵심 이벤트 타입:

- `ContextEvent`
- `InterventionEvent`
- `OutcomeEvent`
- `FeedbackEvent`

공통 필드:

- `eventId`
- `eventType`
- `occurredAt`
- `timezone`
- `source`
- `correlationId`
- `sessionId`
- `version`

`ContextEvent` 최소 필드:

- `signals`
- `derivedState`
- `triggerReason`

`InterventionEvent` 최소 필드:

- `interventionKind`
- `surface`
- `copyVariant`
- `decisionReason`

`OutcomeEvent` 최소 필드:

- `outcomeKind`
- `completionState`
- `latencySeconds`
- `stateDelta`

`FeedbackEvent` 최소 필드:

- `feedbackKind`
- `sentiment`
- `freeText`

규칙:

- 이벤트는 append-only로 기록한다
- `occurredAt`은 사용자 로컬 timezone 기준으로 해석 가능해야 한다
- 하나의 개입 흐름은 같은 `correlationId`로 묶는다
- 동일 이벤트 중복 기록을 막을 수 있어야 한다

영속 규칙:

- 이벤트는 메인 앱 저장소와 같은 로컬 영속 계층에 저장한다
- 이벤트는 원본 `ConditionRecord` / `SleepRecord`와 참조 연결 가능해야 한다
- `sourceRecordId` 또는 동등한 링크 필드를 둬서 원본과 역추적 가능해야 한다

중복 방지 규칙:

- `eventId`는 고유해야 한다
- 동일 개입 흐름 내에서 `eventType + correlationId + surface + occurredAt bucket` 기준 중복 기록을 막는다
- 재시도 저장은 같은 logical event를 새 결과 없이 중복 append하지 않는다

매핑 규칙:

- 체크인 저장은 최소 하나의 `OutcomeEvent`를 남긴다
- 회복 로그 저장은 최소 하나의 `OutcomeEvent`를 남긴다
- proactive 개입 노출은 `InterventionEvent`를 남긴다
- 개입 이후 반응은 `OutcomeEvent` 또는 `FeedbackEvent`로 이어져야 한다

### 9.6 Temporal Semantics

시간 관련 해석 규칙을 고정한다.

원칙:

- `today`는 사용자 현재 timezone 기준의 로컬 날짜다
- 이벤트 저장 시 absolute timestamp와 local timezone 정보를 함께 보존한다
- 수면 기록의 대표 날짜는 기본적으로 `wakeTime` 기준 날짜다
- timezone 변경이 있어도 과거 이벤트의 local day 해석은 당시 timezone 기준으로 복원 가능해야 한다

추가 규칙:

- 로컬 날짜 경계는 `00:00` 기준이다
- 주간 집계는 로컬 주 시작 기준으로 계산한다
- 기본 주 시작은 월요일로 본다
- 자정 이후 종료된 수면은 전날 취침이더라도 기상 날짜에 귀속한다
- 자정을 넘긴 개입 흐름은 `occurredAt` 기준 날짜로 이벤트를 기록한다
- quiet hours 판단도 사용자 로컬 시간 기준이다
- 동일 세션 내 시간 비교는 UTC 변환 기준으로 일관 처리한다

### 9.7 Psychological Measurement Principles

심리 상태 데이터는 `정답`이 아니라 `상황 의존적 self-report`로 취급한다.

원칙:

- 한 번의 응답으로 사용자를 고정 해석하지 않는다
- 같은 점수라도 맥락에 따라 의미가 달라질 수 있음을 전제한다
- missing data를 부정적 상태로 자동 해석하지 않는다
- 측정 편의를 위해 심리적 정확성을 과장하지 않는다

## 10. 저장소 계약

### 10.1 단일 진실원

현재 메인 사용자 플로우의 사실상 단일 진실원은 `LocalDbService`다.

파일:

- `app/lib/core/services/local_db_service.dart`

열리는 Isar 스키마:

- `ConditionRecord`
- `EmotionRecord`
- `OnboardingDataModel`
- `SleepRecord`

### 10.2 허용된 메인 저장 경로

- 체크인 저장: `SeedingNotifier -> LocalDbService.saveConditionRecord`
- 수면 저장: `SleepNotifier -> LocalDbService.saveSleepRecord`
- 온보딩 저장: `OnboardingRepository -> OnboardingLocalDataSource -> Isar`

현재 제품 기준 비핵심/레거시 저장 경로:

- `CaringNotifier -> LocalDbService.saveEmotionRecord`

위 경로는 현재 메인 제품 기능 확장의 기준으로 사용하지 않는다.

### 10.3 수면 하이브리드 구조

현재 수면은 아래 두 구조가 공존한다.

1. `LocalDbService` 기반 Isar 경로
2. `SleepRecordRepositoryImpl` 기반 Hive 경로

규칙:

- 새 작업은 가능하면 Isar 경로를 우선 기준으로 본다
- Hive 경로는 완전 제거 전까지 파손되지 않게 유지한다
- 수면 저장을 제3의 저장소로 늘리지 않는다

### 10.4 Data Ownership & Write Mode

모델별 쓰기 방식:

- `ConditionRecord`: 기본은 append
- `SleepRecord`: create / update 모두 허용
- `EmotionRecord`: 타임라인 래핑 및 레거시 호환 보조 모델
- `OnboardingDataModel`: 사실상 singleton upsert
- `ContextEvent` 계열 이벤트: append-only

추가 규칙:

- 체크인은 같은 날 여러 번 저장될 수 있다
- 수면 기록은 편집 시 기존 `id`를 유지한다
- 이벤트는 원본 기록을 덮어쓰지 않는다
- 새 기능은 기존 모델 책임을 침범하는 새 핵심 모델을 만들지 않는다

### 10.5 Migration Rules

- 새 기능의 영속 저장은 Isar 기준으로 설계한다
- Hive는 수면 레거시 호환 외 새로운 write path를 받지 않는다
- 저장 구조를 바꾸면 이 문서와 실제 마이그레이션 코드를 함께 갱신한다
- 임시 어댑터나 래핑 계층을 새로운 source of truth로 승격시키지 않는다

### 10.6 Edit / Delete / Reset Rules

수정과 삭제는 아래처럼 해석한다.

- 체크인 수정은 새 append보다 명시 edit가 필요할 때만 허용한다
- 수면 기록은 edit를 정식 지원한다
- 이벤트 로그는 원칙적으로 delete보다 append 정정으로 다룬다
- 사용자 기록 삭제 시 관련 집계와 추천 근거에서도 빠져야 한다

리셋 규칙:

- 세션 리셋은 인증/진입 상태만 초기화할 수 있다
- 데이터 리셋은 체크인/수면/이벤트 기록 삭제를 포함하는 별도 동작이다
- 리셋 동작은 서로 혼동되지 않게 분리한다

### 10.7 Event Persistence Rules

- 이벤트 저장은 기록 저장과 같은 트랜잭션 경계 또는 명확한 후속 write로 처리한다
- 원본 기록 저장이 실패하면 성공 outcome event를 남기지 않는다
- 이벤트 write 실패는 사용자 입력 저장 실패와 같은 것으로 취급하지 않되, 재시도 가능해야 한다
- 이벤트 저장 지연이 있어도 원본 기록 조회를 막지 않는다

### 10.8 Schema & Compatibility Versioning

- 영속 모델 구조가 바뀌면 schema version 또는 동등한 호환 식별자가 함께 올라가야 한다
- 핵심 계산식이 바뀌면 이전 기록을 어떤 규칙으로 읽는지 명시해야 한다
- `ConditionRecord`, `SleepRecord`, `OnboardingDataModel`, 이벤트 모델은 버전 변화 시 해석 규칙을 같이 가진다
- 마이그레이션 없이 의미만 바꾸는 변경은 금지한다

## 11. 로직 계약

### 11.1 홈 상태 계산 입력

입력:

- 최신 `ConditionRecord` 1건
- 최근 7일 `ConditionRecord` 목록
- 오늘 `SleepRecord` 1건

산출 상태:

- `latestCondition`
- `todaySleep`
- `conditionScore`
- `averageEnergy`
- `averageFocus`
- `averageRecovery`
- `averageStress`
- `attentionCount`

### 11.2 홈 점수 공식

```text
((energyScore ?? 5)
 + (focusScore ?? 5)
 + (recoveryScore ?? 5)
 + (11 - (stressScore ?? 5))) / 4 * 10
```

의미:

- 에너지, 집중, 회복은 높을수록 좋다
- 스트레스는 낮을수록 좋다

### 11.3 Attention 규칙

주의가 필요한 condition record 조건:

- `score < 55`
- `stress >= 8`
- `recovery <= 4`

`attentionCount`는 최근 7일 condition 중 이 조건에 걸린 개수다.

### 11.4 홈 제안 규칙

홈은 아래 3개를 계산한다.

1. 상태 헤드라인
2. 제안 문장
3. 메인 CTA 1개

우선순위:

1. 오늘 체크인이 없는가
2. recovery가 낮은가
3. stress가 높은가 / attention flag가 있는가
4. focus가 낮은가
5. sleep 데이터가 없는가
6. 최근 평균 수면이 짧은가
7. 최근 평균 stress가 높은가
8. 아니면 좋은 흐름 유지

규칙:

- CTA는 항상 1개만 메인으로 노출한다
- 동일 우선순위 CTA 2개를 병렬로 보여주지 않는다

예시 CTA:

- `오늘 체크인`
- `회복 로그 남기기`
- `다시 체크인하기`

### 11.5 체크인 기본값과 저장

기본 상태:

- `energyScore = 6`
- `focusScore = 6`
- `recoveryScore = 6`
- `stressScore = 4`

overall score:

```text
(energy + focus + recovery + (11 - stress)) / 4 * 10
```

fallback tag 분기:

- `Drained`
- `Wired`
- `Foggy`
- `LockedIn`
- `Overloaded`
- `Steady`

세부 입력 규칙:

- 상태 태그 최대 3개
- 상황 태그 최대 3개
- 몸 태그 최대 3개

저장 결과는 `ConditionRecord`다.

맵핑:

- `selectedTags -> descriptors`
- `selectedContextTags -> contextSignals`
- `selectedBodyTags -> bodySignals`
- `note -> note`

### 11.6 수면 파생 규칙

기본 상태:

- 전날 밤 23시 ~ 오늘 07시 기준 기본값
- 기본 duration 480분

파생 계산:

- `durationMinutes = wakeTime - inBedTime`
- `sleepCycles = (duration - latency) / 90`
- `isGoldenHour`는 완전한 수면 사이클 근처인지 판단

저장 시 계산:

- `sleepEfficiency = (duration - latency) / duration * 100`
- 음수면 `0`
- 100 초과면 `100`

### 11.7 타임라인 래핑 규칙

타임라인은 내부적으로 모두 `EmotionRecord`로 래핑한다.

`ConditionRecord -> EmotionRecord`

- `timestamp = source.timestamp`
- `emotionLabel = descriptor or fallback`
- `activities = condition tag + context tag`
- `bodySensations = body signals + stress pseudo-tag`

`SleepRecord -> EmotionRecord`

- `timestamp = source.date`
- `emotionLabel = sleep recovery` 또는 `sleep log`
- `activities = sleep tags`
- `bodySensations = sleep signal summary`
- `detailedNote = sleep note summary`

의미:

- 히스토리 UI는 단일 엔트리 타입으로 유지
- 실제 저장은 condition/sleep 별도 모델 유지

### 11.8 주간 요약과 insight

최근 7일 condition/sleep에서 다음을 계산한다.

- condition record count
- stable count
- average condition score
- average energy/focus/recovery/stress
- average sleep hours
- average sleep efficiency
- top signals
- insight

insight 우선순위:

1. 데이터 거의 없음
2. 전체 점수 높고 수면 충분
3. stress 높고 recovery 낮음
4. energy 높고 focus 낮음
5. sleep short
6. sleep efficiency low
7. top signal 존재
8. score very low
9. 기본 안정 흐름

### 11.9 Contextual Sampling Rules

질문과 개입은 아래 우선순위로 샘플링한다.

1. 상태 불확실성이 높은가
2. 지금 묻는 것이 행동 결정에 실제로 도움이 되는가
3. 사용자가 응답할 가능성이 높은 시점인가
4. 최근 비슷한 질문을 이미 했는가

억제 규칙:

- 최근 같은 질문을 짧은 시간 안에 반복하지 않는다
- active focus block 중에는 불필요한 질문을 넣지 않는다
- 캘린더상 바쁜 구간으로 판단되면 질문보다 침묵을 우선한다

### 11.10 Intervention Policy

개입은 아래 규칙을 따른다.

개입 허용 조건:

- 상태 악화 또는 위험 신호가 명확하다
- 지금 행동 하나가 실제로 도움이 된다
- 최근 같은 목적의 개입이 실패하지 않았다

개입 억제 조건:

- quiet hours
- 최근 개입 dismiss
- active focus block
- 이미 더 강한 표면에서 같은 개입을 노출함

기본 budget:

- proactive 개입은 하루 1~2회
- 같은 목적의 개입은 짧은 시간 내 재발송 금지
- 시스템 알림은 인앱 표면보다 더 엄격한 기준을 적용

예외 허용:

- 사용자가 직접 시작한 집중 블록의 중간 체크
- 취침 직전 보호성 개입
- 사용자가 명시적으로 요청한 리마인드

표면 우선순위:

1. 홈 CTA
2. 인앱 배너 / 카드
3. 시스템 알림

에스컬레이션 규칙:

- 사용자가 이미 앱 안에 있으면 알림보다 인앱 표면을 우선한다
- 같은 목적 개입은 더 약한 표면으로 해결되지 않을 때만 더 강한 표면으로 올린다
- 같은 세션 안에서는 같은 개입을 여러 표면에 동시에 띄우지 않는다

디에스컬레이션 규칙:

- 동일 목적 개입이 반복 dismiss되면 다음에는 더 약한 표면만 사용하거나 침묵한다
- 같은 목적 개입이 최근 completed면 후속 알림을 보내지 않는다
- home CTA로 이미 충분히 노출된 제안은 같은 시점 알림으로 중복하지 않는다

안전 우선순위:

1. 안전 대응
2. 사용자의 명시적 수동 입력
3. 집중 블록 / quiet hours 억제
4. 일반 proactive 개입 규칙

안전 대응 규칙:

- 위험 신호가 의심되면 일반 코칭 카피보다 중립적이고 축소된 응답을 우선한다
- 안전 대응은 quiet hours, budget, focus suppression보다 우선할 수 있다
- 안전 대응이 발생한 세션에서는 일반 코칭 개입을 추가로 겹치지 않는다
- 안전 대응은 개인화 성공/실패 실험 대상으로 취급하지 않는다

### 11.11 Outcome Evaluation Rules

개입 결과는 아래 수준으로 구분한다.

- `shown`
- `opened`
- `accepted`
- `completed`
- `dismissed`
- `ignored`
- `snoozed`

상태 변화 평가는 아래 중 하나로 정리한다.

- `improved`
- `unchanged`
- `worsened`
- `unknown`

원칙:

- 클릭만으로 성공 처리하지 않는다
- 완료와 개선을 구분한다
- 명시 피드백이 없어도 outcome은 남겨야 한다

### 11.12 Notification Budget & Quiet Hours

기본 quiet hours는 개인화 전까지 `23:00 ~ 07:00`로 본다.

규칙:

- quiet hours에는 일반 상태 질문을 보내지 않는다
- 수면 보호 개입은 취침 직전 window에서 1회만 허용할 수 있다
- 사용자가 알림을 껐으면 인앱 표면만 사용한다
- 같은 날 반복 dismiss가 누적되면 남은 proactive 개입을 줄인다

### 11.13 Data Freshness & Aggregation Windows

상태 해석에 쓰는 데이터의 신선도 규칙을 둔다.

- 홈의 `latestCondition`은 최근 체크인을 우선 사용한다
- 오래된 체크인만 있고 당일 기록이 없으면 `오늘 상태 미확인`으로 본다
- 회복 상태는 오늘의 수면 기록을 우선 사용한다
- 주간 요약은 최근 7일 rolling window를 기본으로 한다
- `recent 7 days`는 오늘 포함 직전 6일을 더한 로컬 날짜 창이다

원칙:

- 오래된 좋은 기록으로 오늘 상태를 과하게 낙관하지 않는다
- 집계 창이 바뀌면 홈, 타임라인, 개입 규칙이 같은 창을 사용해야 한다

### 11.14 Measurement Reactivity & Burden Rules

측정은 상태를 읽는 동시에 상태를 바꿀 수 있다.

원칙:

- 잦은 질문은 자기관찰 피로와 회피를 부를 수 있다
- 질문 빈도는 데이터 욕심이 아니라 순효용 기준으로 줄인다
- 응답률 하락은 사용자 의지 부족보다 설계 부담 신호로 먼저 본다
- 같은 질문을 반복해 정서적 rumination을 유발하지 않는다

### 11.15 Focus Block Rules

집중 블록은 사용자 선언형 기능으로 다룬다.

기본 규칙:

- 사용자가 시작한 세션만 `active focus block`으로 본다
- active 상태에서는 일반 proactive 질문을 억제한다
- 중간 체크는 1회 또는 매우 제한적으로만 허용한다
- 종료 시 outcome과 짧은 피드백을 남길 수 있어야 한다

목적:

- 몰입을 해치지 않으면서 코칭 데이터를 남긴다
- 사용자 주도 행동과 시스템 개입을 구분한다

### 11.16 Aggregation Exclusion Rules

집계에서 제외해야 하는 데이터는 명확히 정의한다.

제외 대상:

- 사용자가 삭제한 기록
- 저장 실패 상태의 임시 레코드
- 필수값이 없는 불완전 레코드
- 디버그/내부 테스트용 엔트리
- 중복으로 판정된 이벤트
- 의미를 복원할 수 없는 레거시 변환 엔트리
- 온보딩 중 생성된 미완료 draft 데이터

원칙:

- 집계 제외는 원본 삭제와 다르다
- 홈, 타임라인, 인사이트가 같은 제외 규칙을 사용해야 한다
- 제외 규칙 때문에 사용자 체감과 다른 결과가 나오지 않도록 화면 간 해석을 맞춘다

### 11.17 Interpretation Limits

시스템 해석은 아래 한계를 가진다.

- 현재 상태 추정은 맥락적 가설이지 성격 판정이 아니다
- 부정 정서는 병리화하지 않는다
- 긍정 정서를 무조건 좋은 성과로 해석하지 않는다
- 낮은 응답 빈도를 무관심, 저항, 악화로 단정하지 않는다

원칙:

- 해석은 행동 지원에 필요한 수준까지만 한다
- 설명 가능하지 않은 강한 심리 결론은 내리지 않는다

### 11.18 Behavior Change Framing Rules

행동 제안은 가능한 한 작고 구체적이어야 한다.

원칙:

- 모호한 다짐보다 즉시 실행 가능한 한 단계 행동을 제안한다
- 회복 제안은 `쉬어라`보다 구체적 행동으로 번역한다
- 한 번에 여러 과제를 제안하지 않는다
- 행동 단위는 성공 확률이 높고 자기효능감을 보존하는 크기로 자른다

### 12.2 사용 자제 단어

- `물 주기`
- `마음 날씨`
- `운영 원칙`
- `정원 일지`

### 12.3 문장 길이

- 제목: 1줄
- 설명: 1~2줄
- 버튼: 2~4어절

### 12.4 레이아웃 규칙

- `SafeArea` 아래에서 시작
- centered AppBar 최소화
- 스크롤 전 첫 화면을 우선 평가
- 카드 하나의 역할은 하나
- 상태 색상은 일관되게 유지
- 배경은 다크 계열 우선

- 한 번에 하나의 행동만 제안한다
- 죄책감 유발 문구를 쓰지 않는다

- dismiss가 예상되는 모호한 문구보다 명확한 행동 문구를 우선한다

### 12.6 심리 안전 카피 규칙

- 평가받는 느낌보다 이해받는 느낌을 우선한다
- `게으름`, `의지 부족`, `망침` 같은 도덕화 표현을 금지한다
- 사용자를 고정 성격으로 규정하는 라벨을 피한다
- 확신이 낮은 해석은 단정형이 아니라 제안형 문장으로 쓴다
- 긍정 피드백도 과장된 칭찬보다 현실적인 효능감 언어를 우선한다

## 13. 구현 원칙

### 13.1 Spec first

코드보다 먼저 이 문서를 확인한다.

이 문서와 어긋나는 코드는 수정 대상이다.

### 13.2 One route at a time

작업은 가능한 한 라우트 단위로 자른다.

예시:

- 홈 단순화
- 체크인 상세 입력 정리
- 회복 로그 카피 정리
- 타임라인 카드 통일

### 13.3 First visible state matters

항상 스크롤 전 첫 화면을 기준으로 평가한다.

질문:

- 첫 3초 안에 화면 목적이 읽히는가
- 동시에 판단해야 할 블록이 2개를 넘지 않는가
- 헤더가 안전영역 아래에서 시작하는가

### 13.4 라우트별 작업 규칙

홈:

- 메인 상태 카드 외에 동급 히어로 블록을 만들지 않는다
- CTA는 1개만 메인으로 둔다
- 타임라인과 중복되는 리포트는 홈에서 줄인다

체크인:

- 기본 입력만으로 저장 가능해야 한다
- 상세 입력은 기본적으로 접혀 있어야 한다
- 점수 계산 로직은 provider에 두고 UI에서 재계산하지 않는다

회복 로그:

- 과학 설명보다 입력 효율을 우선한다
- 저장 후 홈/타임라인 반영이 자연스러워야 한다

타임라인:

- condition/sleep를 하나의 흐름처럼 보여준다
- 카드 구조는 통일하되 라벨만 구분한다

설정:

- 시스템 설정성 항목만 둔다
- 제품 철학을 길게 설명하지 않는다

### 13.5 작업 시작 체크리스트

1. 이 작업이 어느 라우트에 속하는가
2. 이 화면의 목적은 무엇인가
3. 제거해야 할 레거시 메타포가 있는가
4. 변경이 데이터 계약을 건드리는가
5. 저장 구조에 영향이 있는가

### 13.6 Definition of Done

1. 해당 라우트의 화면 계약을 어기지 않는다
2. 카피가 현재 톤과 맞는다
3. 상단 안전영역을 침범하지 않는다
4. `dart analyze` 기준 새 에러를 만들지 않는다
5. UI를 바꿨다면 실제 첫 화면을 확인한다

### 13.7 문서 업데이트 규칙

아래 중 하나가 바뀌면 이 문서도 같이 갱신한다.

- 라우트 표면
- 화면 계약
- 데이터 계약
- 저장 구조
- 메인 카피 체계

문서 없는 구조 변경은 허용하지 않는다.

### 13.8 Verification Rules

- 상태 계산 로직이 바뀌면 최소 provider 단위 검증이 필요하다
- 저장 경로가 바뀌면 생성/수정/조회 흐름을 함께 확인한다
- 개입 규칙이 바뀌면 중복 노출과 quiet hours 위반 여부를 점검한다

### 13.9 Release Gates

릴리스 전 최소 확인 항목:

1. 앱이 `/dev`로 시작하지 않는다
2. 온보딩 가드가 메인 라우트에서 일관되게 동작한다
3. 알림 없이도 핵심 루프가 동작한다
4. 권한 거부 상태에서도 앱이 막히지 않는다
5. 수면/체크인/타임라인의 저장과 재조회가 같은 source를 바라본다

## 14. 레거시 경계와 기술 부채

현재 코드베이스에는 2개의 층이 공존한다.

1. `FullCon 현재 구조`
2. `Mind-Gardener 레거시 구조`

원칙:

- 메인 라우트에 안 붙어 있으면 우선순위를 낮춘다
- 참조가 남아 있으면 먼저 분리 계획을 세운다
- 새 핵심 플로우에 억지로 재활용하지 않는다

현재 메인 제품에서 제거할 것:

- garden 세계관을 제품 중심 은유로 쓰는 것
- plant assignment를 사용자-facing 핵심 기능으로 쓰는 것
- quest / growth / level 구조
- caring / anger를 독립 기능 축으로 유지하는 것

현재 제품에서 남길 것:

- `EmotionRecord`의 타임라인 래핑 역할
- 레거시 데이터 필드를 읽기 위한 최소 호환 계층
- 디버그/마이그레이션을 위한 내부 참조 경로

흡수해서 남길 것:

- 사후 반응 확인 흐름은 `Outcome / Feedback` 유즈케이스로 흡수
- 고스트레스 대응은 `Intervention Policy` 안의 de-escalation 흐름으로 흡수

정리 우선순위:

1. 수면 저장 구조 단일화
2. `EmotionRecord` 히스토리 래핑 구조 재평가
3. 레거시 네이밍 정리
4. caring / anger / garden 모듈 제거 또는 내부 호환 계층으로 축소
5. 인증/세션 구조 현실화

허용 범위:

- 레거시 모듈은 참조 호환 때문에 남을 수 있다
- 하지만 새 핵심 기능의 기반으로 확장하지 않는다
- 새 문서와 새 카피에서 레거시 세계관을 다시 메인으로 끌어오지 않는다

- 룰 기반

중기:

- 패턴 기반

후기:

- 구조화된 데이터와 피드백 루프를 먼저 만든다

## 16. 외부 데이터와 프라이버시 원칙

| 요소 | 역할 |
| --- | --- |
| 체크인 | 내부 상태 |
| 수면 | 회복 상태 |
| 폰 사용 | 행동 패턴 |
| 워치 | 생리 상태 |
| 캘린더 | 상황 맥락 |

특히 캘린더는 `일정을 보여주기 위한 기능`이 아니라 `행동 타이밍을 판단하기 위한 맥락 입력`으로 본다.

### 16.1 사용 목적 제한

- 외부 데이터는 개입 타이밍과 상태 추정을 위해서만 사용한다
- 사용 목적이 설명되지 않는 수집은 허용하지 않는다
- 제품 가치와 직접 연결되지 않는 민감 신호는 기본적으로 수집하지 않는다

### 16.2 보관과 삭제

- 기본 원칙은 local-first다
- 사용자가 삭제한 기록은 관련 화면과 집계에서 즉시 빠져야 한다
- 세션 초기화와 기록 삭제는 다른 동작으로 취급한다
- export / sync / cloud retention은 별도 설계 전까지 기본 가정에 넣지 않는다

### 16.3 신뢰 보호 규칙

- 개입 근거를 설명할 수 없는 블랙박스 수집을 피한다
- 워치/폰/캘린더 신호는 원시 로그보다 요약 신호를 우선 사용한다
- 개인정보 권한이 없어도 핵심 체크인 루프는 동작해야 한다

### 16.4 Permission Consent Principles

- 권한은 필요한 시점에만 요청한다
- 한번 거부된 권한을 같은 세션에서 반복 압박하지 않는다
- 권한이 주는 이익을 행동 단위로 설명한다
- 권한 수락을 개인화의 전제조건처럼 보이게 하지 않는다

## 18. 성공 기준

1. 사용자가 한 번이라도 개입이 정확했다고 느낀다
2. 제안이 실제 행동 변화로 이어진다
3. 시간이 지날수록 개인화가 체감된다

### 18.1 운영 지표

최소로 봐야 하는 지표:

- 체크인 완료율
- 회복 로그 완료율
- 제안 CTA 수락률
- 개입 dismiss 비율
- 개입 후 행동 완료율
- 주간 재방문 또는 루프 유지율

좋은 방향의 신호:

- 질문 수는 줄고 정확도는 오른다
- proactive 개입 수는 적지만 행동 전환율은 높다
- 동일 사용자에게 맞는 개입 패턴이 분화된다

## 19. 제품 진화 단계

### Level 1. Reactive

- 현재 상태 기반 반응

### Level 2. Proactive

- 타이밍 기반 개입

### Level 3. Predictive

- 미래 상태와 위험 타이밍 예측

## 20. 로드맵

### Phase 1. Core Loop

- 실행형 CTA
- 취침 개입
- Micro Check-in
- 기본 피드백

완료 기준:

- 홈 CTA가 실제 행동 흐름으로 연결된다
- 체크인과 회복 로그 후 결과 기록이 남는다
- 최소 1개의 proactive 개입이 규칙 기반으로 동작한다

### Phase 2. Closed Loop

- Outcome tracking
- Feedback 시스템
- 데이터 구조 정리

완료 기준:

- `ContextEvent`, `InterventionEvent`, `OutcomeEvent`, `FeedbackEvent`가 연결된다
- 개입 성공/실패를 회고 없이도 구분할 수 있다
- 중복 저장과 소스 오브 트루스 경계가 정리된다

### Phase 3. 개입 정확도 향상

- Context trigger 확장
- 폰 사용 데이터 활용
- 패턴 기반 개입

완료 기준:

- 타이밍 기반 억제와 허용 규칙이 실제로 적용된다
- 같은 사용자에게 무의미한 개입 반복이 줄어든다
- 상황 기반 질문의 빈도가 낮고 적중률이 높다

### Phase 4. 예측

- 상태 예측
- 위험 타이밍 탐지

완료 기준:

- 미래 위험 구간을 사전 표지할 수 있다
- 사전 개입이 사후 회복보다 더 높은 가치를 만든다

### Phase 5. 확장

- 워치 연동
- 캘린더 연동
- 고급 개인화

완료 기준:

- 외부 신호가 개입 정확도를 실제로 개선한다
- 사용자별 리듬 모델이 분명히 분화된다
- 외부 데이터가 없어도 핵심 루프는 유지된다

## 21. 현재 단계 판정

현재 FullCon은 아래 단계에 있다.

- 아이디어 단계 아님
- 단순 기능 정의 단계 아님
- `제품 시스템 설계 단계`

따라서 지금은 화면 추가보다 `루프 설계`, `이벤트 구조`, `개입 구조`를 우선해야 한다.

현재 우선 공백:

- runtime route guard를 릴리스 기준으로 고정하기
- event schema를 실제 저장 구조로 내리기
- intervention budget과 quiet hours를 코드 규칙으로 연결하기
- 수면 저장소 이원화를 정리하기
