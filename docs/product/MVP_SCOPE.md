# FullCon MVP 현실화 정리

작성일: 2026-05-01  
목표: FullCon을 `기록 앱`이 아니라 `지금 필요한 행동 1개를 제안하는 로컬 우선 컨디션 코치`로 닫는다.

원칙:

- 이 문서는 현재 FullCon 정본 스펙을 MVP 실행 범위로 좁힌 파생 문서다
- 규칙 충돌 시 `current-spec/00-master-spec.md`를 우선한다
- 레거시 ZestInMe / Mind-Gardener 요소는 현재 FullCon 메인 제품 기준으로만 남긴다

## 1. MVP 한 줄 정의

FullCon MVP는 사용자가 빠르게 컨디션과 회복 상태를 남기고, 홈에서 지금 필요한 행동 1개를 받고, 그 결과가 다음 판단에 반영되는 로컬 우선 컨디션 코치 앱이다.

핵심은 `기록량`이 아니라 `현재 상태 파악 -> 행동 제안 -> 결과 기록`의 최소 루프를 안정적으로 닫는 것이다.

## 2. MVP 핵심 루프

MVP에서 반드시 완성해야 하는 루프는 아래 6개다.

```text
1. 첫 실행 온보딩 진입
2. 닉네임 입력 + 첫 컨디션 체크인 저장
3. 홈에서 오늘 상태 / 지금 할 일 / 이번 주 한 줄 확인
4. 오늘 체크인 또는 회복 로그 저장
5. 홈 메인 CTA 또는 최소 1개의 규칙 기반 개입에 반응
6. 타임라인에서 condition + sleep 기록 다시 확인
```

이 6개가 로컬 데이터만으로 안정적으로 돌면 MVP로 본다.

## 3. MVP 포함 기능

### 3.1 앱 시작과 온보딩

반드시 포함:

- 일반 실행 시 `/dev`가 아니라 `/` 또는 `/onboarding`으로 시작
- 온보딩 미완료 사용자는 홈보다 온보딩이 우선
- 닉네임 입력
- 첫 컨디션 체크인 저장
- 첫 체크인 저장 이후에만 온보딩 완료 처리
- 중단 후 재진입 시 마지막 안전 단계부터 복구

보류:

- 환경값 기반 식물 배정
- 복잡한 모듈 선택
- 감성 장면 중심 온보딩 확대
- 다단계 프로필 설문

### 3.2 홈

반드시 포함:

- `오늘 상태`, `지금 할 일`, `이번 주 한 줄` 3개 핵심 블록
- 메인 CTA 1개만 노출
- 최신 `ConditionRecord`, 오늘 `SleepRecord`, 최근 7일 기록 기반 상태 계산
- 오늘 체크인이 없으면 `오늘 체크인` 우선
- 회복 데이터가 없거나 낮으면 `회복 로그 남기기` 우선
- 데이터가 부족하면 empty 상태와 첫 행동 CTA 노출

보류:

- 동시에 여러 CTA 노출
- 대시보드형 다중 인사이트 카드
- 발견/탐색 탭 확장
- 장문 설명형 AI 브리핑

### 3.3 컨디션 체크인

반드시 포함:

- 사용자가 30초 안에 기본 체크인을 저장 가능
- 최소 입력은 `에너지`, `회복`
- 선택 입력으로 `집중`, `스트레스`, `태그`, `상황`, `몸 반응`, `메모`
- 저장 중 상태와 저장 성공/실패 피드백
- 저장 후 홈 데이터 즉시 갱신
- `ConditionRecord` 저장

보류:

- 장문 상담형 입력
- 복잡한 심리 모델 해석 UI
- 진단처럼 보이는 점수화 문구

### 3.4 회복 로그

반드시 포함:

- 취침 시간 입력
- 기상 시간 입력
- 회복감 또는 수면 질 입력
- 메모 또는 태그 입력
- cross-day 시간 정규화
- 기존 기록 수정
- 저장 후 현재 화면 종료 + 홈/타임라인 반영
- `SleepRecord` 저장

보류:

- 고급 수면 효율 분석 리포트
- 복잡한 차트 상호작용
- 외부 수면 기기 연동

### 3.5 타임라인

반드시 포함:

- 월간 캘린더에서 기록 존재 여부 표시
- 특정 날짜 선택 시 condition + sleep 기록을 한 흐름으로 조회
- 최근 7일 기준 주간 한 줄 요약
- 데이터 부족 시 low-confidence 표현 사용
- 기록 수정/삭제 후 집계 재계산

보류:

- 통합 인사이트 리포트
- 긴 회고 코칭 플로우
- 감성형 월간 정원 시각화

### 3.6 최소 개입 시스템

반드시 포함:

- 홈 메인 CTA를 첫 개입 표면으로 사용
- 최소 1개의 규칙 기반 개입 흐름이 동작
- 개입 결과를 `completed`, `dismissed`, `ignored`, `snoozed` 수준으로 기록
- 같은 목적 개입은 dismiss 또는 수동 행동 이후 줄어들어야 함
- quiet hours에는 일반 proactive 질문 억제
- 개입 결과가 이후 판단에 반영될 수 있도록 이벤트 저장

보류:

- 다수의 동시 개입 시나리오
- 고도화된 개인화 모델
- 장문 AI 대화 코칭
- 외부 센서 기반 예측 개입

### 3.7 데이터와 저장소

반드시 포함:

- 온보딩, 체크인, 회복 로그, 이벤트는 로컬 우선 구조 유지
- 체크인/수면/타임라인 재조회가 같은 source를 바라봄
- 수면 저장 경로는 하나로 통일
- 집계에서 삭제, 중복, 미완료 draft 제외
- 외부 권한이 없어도 핵심 루프 동작

권장 기준:

- 온보딩: Isar 또는 현재 로컬 주 저장소 1개
- 체크인: Isar
- 수면: Isar 단일 write path
- 이벤트: 로컬 저장 우선

정리 원칙:

- Hive는 수면 레거시 호환 또는 마이그레이션 대상으로만 둔다
- 새 write path를 추가하지 않는다
- Firebase / 외부 동기화는 MVP 핵심 의존성으로 두지 않는다

## 4. MVP에서 제거하거나 숨길 것

아래 항목은 코드가 일부 남아 있어도 MVP 메인 플로우에서 적극 노출하지 않는다.

- `/dev` 초기 진입
- 로그인 화면
- Mind-Gardener 세계관 복원
- 식물 종 배정 결과 노출
- 대표 식물 외의 다종 식물 UI
- GardenRepository 중심 확장 설계
- 발견/인사이트 탭 확장
- Self-Mirror
- Ventilation
- 소셜 공유
- 서버 백업
- Firebase Auth 의존 플로우
- 장기 개인화 AI 상담
- 워치 / 폰 사용 / 캘린더 외부 데이터 의존 플로우

레거시 식물/정원 자산은 남겨둘 수 있지만, 현재 MVP 정의의 중심 기능으로 다루지 않는다.

## 5. MVP 이후로 미루는 기능

아래는 FullCon 방향에는 맞지만 MVP 1차에서는 후순위다.

- micro check-in 정교화
- 취침 전 wind-down 개입
- 집중 블록 타이머와 중간 확인
- 알림 기반 다중 표면 개입 확장
- 개입 빈도 사용자 세부 조절
- 패턴 기반 개인화 강화
- 외부 데이터로 개입 정확도 높이기

## 6. 우선순위 작업 상태

### 완료된 정리

아래 항목은 FC-001 기준으로 메인 플로우 정렬이 끝났거나 기준선이 잡혔다.

- 앱 진입 플로우 정리
  - 일반 실행 시 `/dev`가 아니라 온보딩 또는 홈으로 진입
  - 온보딩 완료 전 메인 제품 라우트 진입 차단
- 수면 저장소 일원화
  - 메인 write path를 `Isar`로 고정
  - `Hive`는 레거시 코드/정리 대상으로만 남김
- 홈 단일 CTA와 상태 계산 정리
  - 메인 CTA 1개만 유지
  - `오늘 체크인` 우선 규칙을 실제 오늘 기록 기준으로 반영
- 체크인/회복 로그 저장 UX 안정화
  - 회복 로그 저장 성공/실패를 UI에서 구분
  - 홈 회귀 테스트 일부 추가

### 남은 우선순위

#### 1순위: simulator 기준 첫 실행 QA

목표:

- 실제 첫 실행이 `풀컨디션 유지` 약속으로 읽히는지 검증

완료 기준:

- 온보딩 -> 홈 -> 체크인 -> 회복 로그 -> 타임라인 흐름 확인
- 첫 화면 약속, 단일 CTA, 결과 기록 흐름 문서화
- `flutter analyze` / `flutter test` 전체 기준선 상태 기록

#### 2순위: 온보딩 재진입 복구와 원래 요청 라우트 복귀

목표:

- 온보딩이 끊겨도 마지막 안전 단계부터 이어지고, 가로막힌 메인 라우트 요청이 있으면 완료 후 복귀

완료 기준:

- 중단 후 재실행 시 마지막 안전 단계 복구
- `/sleep`, `/history`, `/settings` 진입 요청 복귀
- 관련 라우팅/온보딩 테스트 추가

#### 3순위: 최소 개입 결과 기록

목표:

- 개입이 노출만 되고 끝나지 않게 최소 결과 이벤트를 남긴다

완료 기준:

- 최소 1개의 규칙 기반 개입 흐름 존재
- `completed` / `dismissed` / `ignored` / `snoozed` 중 하나 기록
- 같은 목적 개입의 재노출 억제 규칙 반영

#### 4순위: 타임라인과 집계 재계산 안정화

목표:

- 기록 수정/삭제 이후에도 홈과 타임라인 집계가 흔들리지 않게 한다

완료 기준:

- 날짜별 기록 조회 정상 동작
- 수정/삭제 후 집계 갱신
- 삭제/중복/draft 제외 규칙 일관성 유지

#### 5순위: 전역 품질 기준 정리

목표:

- 전체 `flutter analyze` / `flutter test` 기준을 roadmap 종료 판단에 쓸 수 있게 정리

완료 기준:

- failing suite triage 문서화
- legacy warning/error 정리 또는 분리 기준 확정
- FullCon 메인 루프 검증을 막는 blocker 축소

## 7. MVP 완료 기준

아래 조건을 만족하면 MVP 1차 완성으로 본다.

- 새 사용자가 앱을 켜고 온보딩에서 닉네임 입력과 첫 체크인을 마칠 수 있다
- 홈에 `오늘 상태`, `지금 할 일`, `이번 주 한 줄`이 보인다
- 홈 메인 CTA는 1개만 노출된다
- 사용자가 컨디션 체크인을 저장하고 즉시 홈에 반영되는 것을 본다
- 사용자가 회복 로그를 저장/수정하고 홈 또는 타임라인에서 다시 확인할 수 있다
- 최소 1개의 규칙 기반 개입 흐름이 동작하고 결과 이벤트가 남는다
- 타임라인에서 특정 날짜의 condition + sleep 기록을 같이 확인할 수 있다
- 일반 실행 시 `/dev`로 시작하지 않는다
- 외부 권한이 없어도 핵심 루프가 막히지 않는다
- `flutter analyze`와 핵심 테스트가 통과한다

## 8. 명확히 하지 않을 것

MVP에서는 FullCon을 `완성된 AI 코치`처럼 보이게 만드는 데 시간을 쓰지 않는다.

하지 않을 것:

- 감성 세계관 복원 작업
- 식물 asset 대량 제작
- 정원 중심 브랜딩 확장
- 장면 연출용 애니메이션 확대
- 장문 AI 상담
- 복잡한 예측 모델 마케팅
- 외부 데이터 연동 없이는 못 쓰는 구조

먼저 `체크인`, `회복`, `홈 CTA`, `최소 개입`, `타임라인`이 정확하게 이어지는 구조를 닫는다.
